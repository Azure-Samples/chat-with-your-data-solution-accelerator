- **Status**: Accepted
- **Date**: 2026-06-20
- **Phase**: 6 (Functions blueprints / modular RAG indexing pipeline; `BUG-0054`)
- **Pillar**: Stable Core (the blob-ingestion trigger contract)
- **Deciders**: CWYD v2 maintainers (repo owner)
- **Companion**: [ADR 0002](0002-no-key-vault-uami-rbac.md) (UAMI + RBAC, no shared keys â€” why Event Grid delivers with a managed identity); references `BUG-0054`, `BUG-0056`, `BUG-0053`, `BUG-0058`
- **Revised**: 2026-06-20 â€” the delivery mechanism changed from an Event Grid *trigger* (AzureFunction destination) to a dedicated `blob-events` Storage Queue + a queue-trigger translator (option **B1**). The single-trigger architecture (Event Grid as the one ingestion trigger; admin upload writes the blob only) is **unchanged**; only the wire from the topic to the function changed. The Event Grid *trigger* was rejected because an `AzureFunction` destination (a) does not support managed-identity delivery â€” Event Grid fetches the function's *access key* â€” and (b) requires the function to already exist so its key can be fetched, which fails the `azd` provision-before-deploy order. See the Decision + Alternatives.

## Context

The v2 infrastructure deploys an Event Grid **system topic** (`evgt-<SUFFIX>`) on the storage account with one subscription, `blob-created-to-doc-processing`, that forwards every `Microsoft.Storage.BlobCreated` event under `/documents/` into the `doc-processing` queue ([main.bicep](../../infra/main.bicep)). The intent was "drop a file in the container â†’ it auto-ingests."

It never worked. The only consumer of `doc-processing` is the `batch_push` queue trigger, which validates a CWYD ingestion envelope (`BatchPushQueueMessage`: `container_name`, `filename`, `ingestion_job_id`, `force_reindex`). An Event Grid `BlobCreated` event is a completely different schema, so every event fails validation, retries, and poisons (`BUG-0054`). **No translator from `BlobCreated` â†’ CWYD envelope was ever written.**

Two reference accelerators were checked for a pattern to adapt:

- **MACAE (Multi-Agent Custom Automation Engine)** has **no** Event Grid ingestion at all â€” no system topic, no blob trigger, no queue ingestion. It seeds data at **provision time** via a script (`infra/scripts/index_datasets.py`) that reads blobs and writes straight to AI Search.
- **CWYD v1** *does* drive ingestion from Event Grid, but via a **polymorphic consumer**: a single queue trigger (`batch_push_results`) on `doc-processing` that branches on the message's `eventType` and handles `Microsoft.Storage.BlobCreated` / `BlobDeleted` events **directly** (pulling the filename out of `data.url`) alongside the legacy `BatchStartProcessing` envelope (`eventType == ""`). So v1 needed no separate translator â€” its *consumer* understood the raw Event Grid schema. (An earlier draft of this ADR wrongly stated v1 was HTTP-driven with no Event Grid; corrected 2026-06-20 after reading `code/backend/batch/batch_push_results.py`.)

v2 carried the Event Grid â†’ `doc-processing` subscription over from v1's shape but **deliberately did not** carry over v1's polymorphic loose-dict consumer: v2's `batch_push` validates a single strict, frozen `BatchPushQueueMessage` (`extra="forbid"`), so a raw `BlobCreated` event is a schema it rejects. The events therefore had no handler that understood them and poisoned. Reusing v1's polymorphic-consumer pattern is off the table â€” the v2 rebuild does not imitate v1's untyped, `eventType`-branching style â€” so B1 instead adds a small, typed translator that keeps `batch_push` single-schema. Document ingestion already works without the Event Grid path (admin upload and `add_url` both enqueue / ingest directly).

The deciding product fact: operators **bulk-upload by dropping files directly into the `documents` blob container** (Storage Explorer, `azd`-seeded sample data, other tooling) â€” a workflow that bypasses the admin API entirely. Those files only ever fire `BlobCreated`; with no handler they are **never ingested**. So the Event Grid path is genuinely needed, not redundant â€” it just needs the missing handler built.

An encoding fact shapes the mechanism. The Azure Functions queue trigger expects **base64** message bodies by default; the v2 `batch_start` producer writes **raw JSON** (no encode policy), so `BUG-0056` set the host-wide `extensions.queues.messageEncoding = none`. Event Grid delivers to a Storage Queue as **raw (non-base64) JSON** â€” which is precisely why `messageEncoding = none` is the standard configuration for an Event-Gridâ†’Storage-Queueâ†’Functions pipeline, the configuration this deployment already runs. So a queue that Event Grid writes and a Functions queue trigger reads under `messageEncoding = none` works with **no** encoding change; the genuine `BUG-0054` failure was the *schema* mismatch (a `BlobCreated` event is not a `BatchPushQueueMessage`), not encoding. The `blob_event` translator additionally tolerates either raw or base64 event bodies defensively, since the exact encoding is an Event Grid behavior outside our control.

## Decision

1. **Build the missing handler as a queue-trigger translator behind a dedicated `blob-events` Storage Queue (option B1).** A new Functions blueprint `blob_event` (`v2/src/functions/blob_event/`) is a `@bp.queue_trigger` on a new `blob-events` queue. The Event Grid system topic delivers `BlobCreated` events to `blob-events` (a `StorageQueue` destination, the same managed-identity delivery the topic already uses). The translator reads the raw Event Grid event, extracts the blob `subject`, builds a `BatchPushQueueMessage`, and enqueues it to `doc-processing` via the existing `enqueue_push_message` producer. `batch_push` is **unchanged**; `doc-processing` stays single-schema, single-encoding (raw-JSON CWYD envelopes only). This keeps the no-keys managed-identity posture ([ADR 0002](0002-no-key-vault-uami-rbac.md)) and deploys as pure bicep, because the `blob-events` queue exists at provision time (no function-existence dependency).

2. **Event Grid becomes the single trigger for blob ingestion (clean, no double-processing).** Any blob written to `documents/` â€” by bulk drop, admin upload, or anything else â€” is ingested through exactly one path (Event Grid â†’ `blob-events` â†’ `blob_event` â†’ `doc-processing` â†’ `batch_push`). Consequently the backend admin upload (`backend.services.ingestion.upload_document`) is reduced to **write-the-blob-only**: its direct `doc-processing` enqueue is removed, because the blob write itself now triggers ingestion via Event Grid. The backend no longer needs to know the queue for uploads.

3. **`batch_start` / reprocess stays as the explicit "reprocess everything" admin action.** It enqueues *existing* blobs (it does not write new blobs), so it never double-fires Event Grid. `add_url` is unaffected (it fetches a URL and indexes inline; no blob is written).

4. **Repoint the subscription; delete nothing.** The bicep subscription destination changes from the `doc-processing` queue to the new `blob-events` queue (still a `StorageQueue` destination with managed-identity delivery, `deliveryWithResourceIdentity`, per [ADR 0002](0002-no-key-vault-uami-rbac.md), since `allowSharedKeyAccess = false`). The system topic and its `BlobCreated` + `documents/` filter are kept; the account-scoped Storage Queue Data Message Sender role assignment is kept (it covers `blob-events`). A `blob-events` (+ `blob-events-poison`) queue is added.

5. **Rollout order preserves a working pipeline at every step.** Land the `blob_event` handler + the bicep repoint **first** (Event Grid starts working; the admin upload still enqueues, so a brief, harmless idempotent double-ingest during rollout), validate live, **then** remove the backend enqueue **last** (eliminating the double â†’ clean single-trigger). The backend enqueue is never removed before the Event Grid path is live.

## Consequences

- **+** Bulk-drop ingestion works: a file dropped directly into the `documents` container is ingested with no admin-API call â€” the operator workflow is supported.
- **+** Exactly one ingestion path for all blobs â€” no double-processing, no dedup heuristics, no two code paths doing the same job. The backend is decoupled from the `doc-processing` queue for uploads.
- **+** The dedicated `blob-events` queue keeps `doc-processing` single-schema (only CWYD envelopes); `batch_push` is untouched. Managed-identity delivery is preserved (no function keys), and the change is pure bicep (the queue exists at provision time â€” no function-existence ordering dependency).
- **+** Nothing is deleted â€” the deployed system topic + subscription are repointed into a working configuration.
- **âˆ’** The admin upload now depends on Event Grid delivery (asynchronous, the path that was previously broken) rather than a synchronous in-process enqueue. Mitigated by Event Grid's retry policy (30Ă—/24 h) and by `batch_start` remaining as an explicit reprocess safety valve. The upload already returned before ingestion completed (async via the queue), so there is no UX regression.
- **âˆ’** `UploadResponse` changes: its `ingestion_job_id` no longer correlates to the real ingest job (the `blob_event` handler mints its own envelope id), and `queued` shifts meaning to "accepted; Event Grid will ingest". The field is dropped or marked informational; the upload UX polls `GET /api/admin/documents` either way.
- **âˆ’** A new `blob-events` (+ poison) queue and a small queue-trigger translator are added (one extra hop: event â†’ `blob-events` â†’ translate â†’ `doc-processing`). The translator depends on the host `messageEncoding = none` (same as `batch_push`) and additionally tolerates a base64 event body defensively.
- **âˆ’** In the cloud, all queue-driven ingestion â€” existing and new â€” is gated by `BUG-0053` (Flex "scale-from-zero never wakes `batch_push`"); full cloud bulk-ingest may require that to be resolved. Local development is unaffected (the host always polls).

## Alternatives considered

- **B2 â€” Event Grid *trigger* function (`AzureFunction` destination).** Originally chosen, then rejected on closer inspection. An `AzureFunction` destination (a) does **not** support managed-identity delivery â€” Event Grid fetches the function's *access key* (per the [Event Grid functions-handler docs](https://learn.microsoft.com/en-us/azure/event-grid/handler-functions)), deviating from the no-keys posture ([ADR 0002](0002-no-key-vault-uami-rbac.md)); and (b) requires the target function to **already exist** so Event Grid can fetch its key at subscription-creation time, which fails the `azd` lifecycle (infra is provisioned *before* code is deployed) and would force a `postdeploy` `az eventgrid ...` script. The chosen B1 queue destination has neither problem (the queue exists at provision; delivery stays managed-identity).
- **C â€” MACAE-style provision-time batch indexing.** A post-provision script ingests the `data/` folder after deploy and leaves the Event Grid subscription idle (not deleted). This is what MACAE literally does, but it does **not** deliver runtime bulk-drop auto-ingest (the operator's actual need) â€” a file dropped after provisioning would still never ingest. Rejected as the primary mechanism; could complement B1 for first-boot seeding.
- **2c â€” accept idempotent double-ingestion.** Keep both the admin enqueue and the Event Grid handler; admin-UI uploads ingest twice (overwriting the same doc id, so the result is correct) at double embedding cost. Rejected: leaves a permanent redundant path and avoidable cost â€” counter to the single-clean-path goal.
- **Delete the Event Grid subscription.** The minimal cleanup if blob-drop auto-ingest were unwanted. Rejected: the operator uses bulk upload, so the feature is needed, not removable.

## Follow-ups

- **`BUG-0056` encoding back-port â€” landed in `host.json`.** `host.json` now sets `extensions.queues.messageEncoding = none`, so every deploy ships it and both `batch_push` and `blob_event` decode the raw-JSON event from their queues correctly; the translator additionally tolerates a base64 event body defensively. The live `AzureFunctionsJobHost__extensions__queues__messageEncoding = none` app-setting override (applied by hand under `BUG-0056`) is therefore redundant once the current package deploys, and no bicep app-setting parity is required because the package-level `host.json` is authoritative.
- **`BUG-0053` (cloud scale-from-zero).** Tracked separately; gates cloud bulk-ingest end-to-end.
- **`BUG-0058` (stale `build-functions/` artifact).** The function redeploy that ships `blob_event` must run `prepackage` first, or it will deploy a stale artifact without the new blueprint.
