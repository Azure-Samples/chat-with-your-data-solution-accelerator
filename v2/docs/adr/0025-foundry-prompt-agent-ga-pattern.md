# ADR 0025 — Foundry **Prompt Agent** GA pattern: name-addressed identity, server-side KB grounding, client-side `Agent` invocation, unified RAI + shared citations

- **Status**: Accepted
- **Date**: 2026-06-12
- **Phase**: 8 (agent runtime — `BUG-0021` structural fix; recorded during the Phase 6 window)
- **Pillar**: Stable Core (agent provisioning + invocation lifecycle, the `build_agent` composition seam, and the shared citation adapter) — adds no new configuration surface beyond the one `AZURE_AI_SEARCH_CONNECTION_NAME` setting that ADR 0021's KB decision already implied
- **Deciders**: CWYD v2 maintainers (repo owner)
- **Amends**:
  - [ADR 0008](0008-lazy-foundry-agent-bootstrap.md) — the lazy-bootstrap-cached-in-DB *principle* is retained, but the agent **create path** changes from `azure.ai.agents.AgentsClient.create_agent` (an Assistants-API `asst_` persistent agent) to `AIProjectClient.agents.create_version` + `PromptAgentDefinition` (a name-addressed Foundry Prompt Agent), and the cached/returned identity changes from the opaque `asst_` id to the agent **name**.
  - [ADR 0021](0021-agent-framework-foundry-iq-kb-default.md) — the KB consume sub-variant flips from **A2** (client-side `agent_framework.MCPStreamableHTTPTool` bound per-run on the by-name `FoundryAgent`) to **A1** (server-side `azure.ai.projects.models.MCPTool` carried on the `PromptAgentDefinition`). ADR 0021 itself documented A1 as "the documented fallback if A2's client-side MCP execution proves insufficient" — it was. ADR 0021's headline decisions (`agent_framework` is the default; grounding is a Foundry IQ KB over a `searchIndex` wrapping `cwyd-index`; KB API version env-pinned) stand unchanged.
- **Companion**: [bugs.md `BUG-0021`](../bugs.md#bug-0021--agent_framework-chat-404s-persistent-assistant-vs-name-referenced-foundry-agent-mismatch) (the defect this fixes), `BUG-0020` (the client-side-bearer `401` fix this pattern retires), `BUG-0022` (dropped system prompt — addressed by the same `build_agent` → `_resolve_definition` instructions flow)

## Context

The `agent_framework` orchestrator (made the default in [ADR 0021](0021-agent-framework-foundry-iq-kb-default.md)) failed in two successive layers during the first live end-to-end exercise against a real Foundry project. Both layers trace to the same architectural seam: **the provisioning half and the invocation half referenced two different Foundry agent resource types.**

1. **`BUG-0020` (`401` on the KB MCP connect).** ADR 0021's spike chose sub-variant **A2** — attach an `agent_framework.MCPStreamableHTTPTool` to the runtime `FoundryAgent` per-run, authenticated through the SDK's `header_provider` hook. That hook is applied only inside `MCPStreamableHTTPTool.call_tool()`; the MCP **session-initialize connect** runs earlier (when the tool's async context manager is entered), so the connect carried no `Authorization` header and Azure AI Search rejected it with `401`. The interim fix threaded a caller-owned `httpx.AsyncClient` with a default bearer header (and owned its lifecycle in a `finally`). It worked, but it pushed transport-credential plumbing into the orchestrator's run loop.

2. **`BUG-0021` (`404 Agent 'cwyd' not found`).** With the connect authenticated, the agent run still `404`ed. Provisioning ([ADR 0008](0008-lazy-foundry-agent-bootstrap.md)'s `get_or_create_agent`) created an **Assistants-API persistent agent** (`asst_` id) via `AgentsClient.create_agent`. Invocation referenced the agent **by name** through the Responses API (`FoundryAgent(agent_name="cwyd")` → `agent_reference {"name": "cwyd"}`), which resolves a Foundry **Prompt/Hosted** agent resource — a *different resource type* than the `asst_` persistent agent that provisioning had created. No name-resolvable `cwyd` resource existed, so Foundry returned `404`. The conversation router compounded the gap by calling `get_or_create_agent` and then **discarding the returned `asst_` id**, passing the friendly name to the orchestrator.

Both defects are facets of the **decoupled hybrid** the dev_plan flagged (`B1-MAF-MISLABEL` / `B-IMPL-AUDIT`): a create path and an invoke path that addressed different resource types and authenticated different surfaces. Patching either layer in isolation leaves the seam misaligned.

The operator's direction (2026-06-11/12 planning) is to align both halves on the **GA Foundry Prompt Agent pattern as implemented by MACAE** — a single name-addressed agent resource, provisioned and invoked through the GA control plane, with KB grounding executed server-side under the Project identity. This is the same pattern the [Multi-Agent Custom Automation Engine](https://github.com/microsoft/Multi-Agent-Custom-Automation-Engine-Solution-Accelerator) uses, adapted (not copied) to CWYD's registry-dispatched orchestrator seam.

## Decision

Adopt the **Foundry Prompt Agent GA pattern** across every agent path (chat and RAI). Concretely, seven aligned decisions:

1. **Name-addressed Prompt Agent provisioning.** `FoundryAgentsProvider.get_or_create_agent(definition, db)` builds a `PromptAgentDefinition(model, instructions, tools)` and calls `AIProjectClient.agents.create_version(agent_name=, definition=, description=)`. The Assistants-API `AgentsClient.create_agent` (`asst_`) path is removed. The abstract provider client type moves from `azure.ai.agents.aio.AgentsClient` to `azure.ai.projects.aio.AIProjectClient` (the GA control plane that owns `agents.create_version` / `agents.get`).

2. **Name-based DB identity.** `get_or_create_agent` returns and persists the agent **name**, not the `asst_` id. A cold start validates any persisted name with `AIProjectClient.agents.get`; a `404` (`ResourceNotFoundError`) is orphan-recovery control flow that falls through to create; a `409` on create is a concurrent-create race that **re-reads and reuses** the existing named agent (idempotent — the conflict is never surfaced as an error). This amends [ADR 0008](0008-lazy-foundry-agent-bootstrap.md): the lazy-create-cache-in-DB lifecycle is unchanged, but the stored identity is now the stable, human-meaningful **name** rather than an opaque generated id, which is also what makes the agent resolvable by other consumers (see §"Multi-agent coordinator readiness").

3. **Server-side KB grounding (consume sub-variant A1).** The KB tool is an `azure.ai.projects.models.MCPTool(server_url=".../knowledgebases/<kb>/mcp", require_approval="never", allowed_tools=["knowledge_base_retrieve"], project_connection_id=<connection_name>)`. Foundry IQ runs the retrieval call **server-side under the Project's `CognitiveSearch` connection identity** — there is no client-side bearer to mint and no `httpx` client for the app to own. The connection name comes from `SearchSettings.connection_name` (`AZURE_AI_SEARCH_CONNECTION_NAME`, wired in Bicep from the `aiProjectSearchConnection` output in `cosmosdb` mode; empty in `postgresql` mode). This **retires the entire `BUG-0020` client-side machinery** — the `header_provider` hook, the caller-owned `httpx.AsyncClient`, and its `finally`-based lifecycle.

4. **Client-side `agent_framework.Agent` invocation (never the by-name `FoundryAgent`).** Invocation composes a client-side `agent_framework.Agent` over an `agent_framework_foundry.FoundryChatClient` bound to the same project endpoint and the agent's own model deployment — mirroring MACAE's `AgentTemplate.open()` composition. The by-name `FoundryAgent` Responses path (the `404` source) is removed. The `Agent` is an async context manager, so the run loop is `async with agent:` (there is no `close()`/`aclose()` on `agent_framework.Agent`). Composing a client-side `Agent` — rather than referencing a server agent by name — is also what keeps the object drivable by a future multi-agent orchestrator (§"Multi-agent coordinator readiness").

5. **One `build_agent` DRY seam for all agent paths.** `BaseAgentsProvider.build_agent(definition, db, *, extra_tools=None) -> Agent` (an `@abstractmethod`) resolves/creates the named Prompt Agent via `get_or_create_agent`, then composes the runtime `Agent`. `extra_tools` are additive per-run tools the caller attaches (the chat orchestrator passes `[kb_tool.as_dict()]`; RAI passes nothing). Both the chat path and the RAI path build their runtime agent through this single seam, so agent lifecycle, instruction resolution, and the SDK-boundary resilience ladder are written once. Instructions flow from `build_agent` → `_resolve_definition` into the runtime `Agent` (no `system_prompt` threading in the orchestrator), which is what closes `BUG-0022` (the dropped-system-prompt gap) by construction.

6. **Unified RAI on the same seam.** `rai_check(text, agents, db) -> bool` (the optional pre-orchestrator safety gate) moves off the Assistants threads/runs loop (`threads.create` → `messages.create` → `runs.create_and_process` → `messages.list`) onto `agent = await agents.build_agent(RAI_AGENT, db)` + a single non-streaming `response = await agent.run(text)` inside `async with agent:`, parsing `response.text.strip().upper().startswith("TRUE")`. The signature is unchanged (callers in [chat.py](../../src/backend/core/pipelines/chat.py) and [admin.py](../../src/backend/services/admin.py) are untouched). RAI **stays on the strong `gpt_deployment`** — an operator quality decision, not downgraded. Per Hard Rule #14 the `agent.run` boundary is wrapped `try/except AzureError` → structured `logger.exception(operation="rai_check", ...)` → **re-raise**: a safety guard must fail *closed by propagation*, never fail *open* by swallowing a transport error into a permissive verdict.

7. **Shared citation seam across orchestrators.** `citations_from_annotations(annotations) -> list[Citation]` ([citations.py](../../src/backend/core/tools/citations.py)) maps `agent_framework` native `Content.annotations` (`type == "citation"`) onto the shared `Citation` model — filtering to citation-type annotations, deriving `id` from `file_id` → `url` → `title`, deduping by `id`, and merging `annotated_regions` spans under `metadata`. The `agent_framework` orchestrator accumulates annotations across the stream, maps them once through this seam, and emits one `OrchestratorEvent(channel=CITATION, ...)` per source **before** the final `answer` — identical wire shape and ordering to the `langgraph` numbered-marker path. Both orchestrators now emit one citation shape on the SSE `citation` channel ([ADR 0007](0007-orchestrator-event-typed-sse-channel.md)), closing the citation-parity gap (`BUG-0013` / `0015` / `0016`) on the `agent_framework` branch.

## Multi-agent coordinator readiness (ISE coordinator-patterns review)

The operator asked whether the ISE post [*Orchestration Patterns for Multi-Agent Systems: Performance and Trade-offs*](https://devblogs.microsoft.com/ise/coordinator-patterns-multi-agent-systems/) suggests changes for CWYD v2. The article documents an ISE + retail-customer engagement evolving a production chatbot from a **router-as-modular-monolith** (deterministic intent routing to exactly one agent, no synthesis) toward a **microservices Coordinator pattern** (coordinators orchestrate multiple independent domain agents, with response synthesis). It weighs three patterns — **Router** (one agent per query), **Group Chat** (shared thread + a Group Chat Manager), and **Coordinator** (chosen) — and offloads cross-cutting concerns (auth, observability, tracing) to an **Azure APIM gateway** between coordinators and domain agents.

The article's **headline lesson** is directly relevant: *both teams prioritized architectural independence between coordinators and domain agents over framework selection* — domain agents are independent, reusable resources behind well-defined interfaces, and the coordinator's framework choice is independent of how each domain agent is implemented.

How this ADR's decisions align, and what is deliberately deferred:

- **CWYD v2 today is the article's *starting point*** — a router-as-modular-monolith. The registry selects exactly one orchestrator per request (`langgraph` *or* `agent_framework`); there is no multi-agent synthesis. That is correct for the current single-tenant RAG product.
- **Framework independence is already structural in v2.** `Registry[BaseOrchestrator]` (Hard Rule #4 / [ADR 0001](0001-registry-over-factory-dispatch.md)) lets a graph framework (`langgraph`) and the Agent Framework (`agent_framework`) coexist behind one async interface (Hard Rule #5). Adding a coordinator is "register one more key," not a rewrite — the exact independence the article argues teams should prioritize.
- **The GA pattern is a concrete step toward reusable domain agents.** Naming each agent (`cwyd`, `rai`) as a server-side Foundry Prompt Agent whose definition lives in Foundry and resolves **by name** means a future coordinator — or a different application — can reuse the same agent without copying its definition. Decisions 1, 2, and 4 (name identity + client-side `Agent` composition over `FoundryChatClient`, not a by-name reference baked into one app) keep the runtime object drivable by an external orchestrator, matching MACAE's group-chat/handoff object shape.
- **Streaming is already the article's perceived-latency mitigation.** v2's typed reasoning-channel SSE ([ADR 0007](0007-orchestrator-event-typed-sse-channel.md)) streams partial results immediately — one of the article's five named mitigations for coordinator overhead.
- **Deliberately deferred (anti-speculation, per [ADR 0024](0024-withdraw-per-tenant-runtime-config-single-tenant.md)'s discipline):** the **Coordinator pattern itself**, **microservices decomposition** of domain agents, an **APIM gateway** for cross-cutting concerns, and **Group-Chat** orchestration. The article is explicit that the coordinator's benefits (reuse, ownership clarity, cross-system composition) come with **real, measured latency overhead** ("no free lunch") that is only worth paying when the organizational driver — agent reuse across many teams and client experiences — is real. CWYD v2 is a single application with a single owner; that driver is absent today. The registry seam + name-addressed agents keep the upgrade path open without paying the coordinator's latency tax now. Reopen when a concrete cross-team agent-reuse requirement lands.

## Consequences

### Positive

- **One agent resource type end-to-end.** Provisioning and invocation reference the same name-addressed Prompt Agent — the `asst_`-vs-name-reference split that caused `BUG-0021` cannot recur.
- **No client-side credential plumbing.** Server-side KB grounding (Decision 3) removes the `header_provider` hook, the caller-owned `httpx.AsyncClient`, and its lifecycle `finally` — the orchestrator's run loop no longer mints or owns a Search bearer (`BUG-0020`'s complexity is gone, not patched).
- **System prompt can no longer be dropped.** Instructions flow through `build_agent` → `_resolve_definition` into the runtime `Agent` by construction (closes `BUG-0022`).
- **DRY across chat and RAI.** One `build_agent` seam owns agent lifecycle, instruction resolution, and the SDK-boundary resilience ladder for both paths; adding an Nth agent is a definitions-file change.
- **Citation parity.** Both orchestrators emit one `Citation` shape on the SSE `citation` channel.
- **Idempotent, race-safe provisioning.** Name validation + `409`-reuse make get-or-create safe under concurrent first requests and across redeploys (steady state: exactly one `cwyd` + one `rai` agent per project).
- **Coordinator-ready without coordinator cost.** Name-addressed, server-defined agents + the registry seam keep the multi-agent upgrade path open while v2 stays a single-orchestrator router.

### Negative

- **Couples the chat path to Foundry IQ KB + the Project search connection.** KB reachability and the `CognitiveSearch` connection's RBAC are now load-bearing for the default path; health/diagnostics must classify KB reachability (`pass | degraded | fail`) per the resilience contract.
- **`agent_framework` + KB remains Azure-AI-Search-only.** Unchanged from [ADR 0021](0021-agent-framework-foundry-iq-kb-default.md): pgvector deployments must use `langgraph`; selecting `agent_framework` there raises a clean `ConfigResolutionError` ([ADR 0022](0022-config-resolution-error-on-incompatible-overrides.md)).
- **Pre-existing `PromptAgentDefinition(tools=...)` type debt is exposed.** `AgentDefinition.tools` is `tuple[str, ...]` (opaque registry keys) while `PromptAgentDefinition.tools` expects `list[Tool] | None`, so `pyright --strict` flags `base.py:258` (`PROMPTDEF-TOOLS-TYPE-DEBT`, [development_plan.md](../development_plan.md) §0.1). Runtime-benign today (both built-ins declare `tools=()`, so the expression is always `None`), but it leaves the backend strict gate RED pending a one-turn operator design call (drop the `tools=` arg vs. add a string-key→`Tool` converter — Hard Rule #10).

### Neutral

- **The DB `agents` shape is unchanged.** Still one row per agent key in the same DB as chat history; only the stored *value* changes meaning (name, not `asst_` id). No new Azure resource.
- **RAI deployment is unchanged.** `RAI_AGENT.deployment_attr` stays `gpt_deployment` (strong model) — the migration changed only the invocation mechanism, not the model.

## Alternatives considered

1. **Patch only the `BUG-0021` invocation (invoke the existing `asst_` agent by id).** Rejected: the OSS `FoundryAgent` exposes no `agent_id` parameter (name + optional version only, for Prompt/Hosted agents), and no `agent_framework` persistent-agent (Azure AI Agent Service threads/runs) client is installed. There is no supported path to invoke an `asst_` agent from the pinned runtime.
2. **Toggle `allow_preview` on `FoundryAgent`.** Rejected: it does not change the resource type the name-reference resolves — both code paths reference a Prompt/Hosted agent by name, so the `404` persists.
3. **Keep client-side A2 and only fix the connect auth (stay on `BUG-0020`'s `httpx` client).** Rejected: it leaves transport-credential plumbing in the run loop and does not address the `BUG-0021` resource-type split at all. Server-side A1 — the fallback ADR 0021 itself documented — removes both problems at once.
4. **Adopt the Coordinator / multi-agent pattern now (per the ISE article).** Rejected as out of scope for the single-tenant deployment (see §"Multi-agent coordinator readiness"): the article documents real latency overhead justified only by cross-team agent-reuse drivers that CWYD v2 does not have today. The registry seam keeps it a future "register one more key," not a rewrite.

## Reopen criteria

Revisit the multi-agent deferral (Alternative 4) only when a concrete requirement to **reuse CWYD's domain agents across multiple teams or client experiences** lands — at which point the Coordinator pattern, microservices decomposition, and an APIM cross-cutting-concerns gateway are evaluated together, with latency measured early per the article's central caution.
