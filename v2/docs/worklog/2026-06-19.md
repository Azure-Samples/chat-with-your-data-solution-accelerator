## Summary

Continued the postgres + pgvector parity validation against the deployed `<AZD_ENV_NAME>` environment, running the v2 app **natively on the workstation against the cloud data plane** (no local Postgres). Created the gitignored Functions host config, enabled Storage public access (RBAC-gated) for the local host, uploaded a sample document, and drove it end-to-end through `batch_start` → `batch_push` into the cloud **pgvector `documents` table** (6 chunks, all 1536-dim). Two findings surfaced: (1) the cloud **PostgreSQL Flexible Server had auto-stopped** (`state=Stopped`), which presents as `asyncpg ConnectionError: unexpected connection_lost() call` during the TLS handshake on every fresh connection — recovered by starting the server; (2) a real **read-side retrieval defect** ([BUG-0065](../bugs.md)): the `langgraph` orchestrator never embeds the query, so `PgVector.search` falls back to `plainto_tsquery` AND-semantics full-text search, which matches no single small chunk for a natural-language question → no citations + the out-of-domain fallback. Ingestion and storage are correct; only the query construction is wrong.

* **Recovered the auto-stopped PostgreSQL Flexible Server.** Every fresh `asyncpg` connection failed with `ConnectionError: unexpected connection_lost() call` at the SSL upgrade — reproduced from a standalone script on both Python 3.13.14 and 3.14.3 (so **not** a Python/asyncpg version issue, and the Functions pool code is byte-identical to the backend pool). ## Decisions

* **Local app, cloud data — no local Postgres.** Per the user, the workstation runs the backend + Functions host natively against the deployed cloud Postgres/Storage/Foundry, gated by `az login` RBAC.
* **Storage opened RBAC-gated, not key-based.** `publicNetworkAccess=Enabled` + `defaultAction=Allow` with no shared-key access — every data call still requires an Entra data-plane role.
* **BUG-0065 fix belongs on the read path, not the FTS branch.** The query must be embedded and passed as `vector=` so `PgVector` does dense cosine retrieval (its intended mode); the FTS branch stays only as a no-embedder fallback. This also upgrades `AzureSearch` to true hybrid. Implementation deferred to the next unit (test-first).
