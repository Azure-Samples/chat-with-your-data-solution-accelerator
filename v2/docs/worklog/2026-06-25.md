### Gotcha 1 — RBAC back-port collision (one-time migration)

Back-porting a previously-manual role grant into Bicep against an env that **already has the manual grant** fails the first redeploy with `RoleAssignmentExists` (the Bicep-declared assignment uses a different deterministic GUID name than the manual one, so ARM sees a duplicate). azd runs incremental ARM, so it never deletes the manual assignment on its own.

- Symptom: `RoleAssignmentExists: The role assignment already exists.` during provision.
- Diagnosis: `az role assignment list --all --query "[?name=='<GUID>']"` showed the colliding grant (here `Cognitive Services User` for the shared UAMI `<AZURE_UAMI_PRINCIPAL_ID>` scoped to the Foundry account).
- Fix: delete the manual assignment so Bicep owns it on the next pass — `az role assignment delete --role "<ROLE>" --assignee <AZURE_UAMI_PRINCIPAL_ID> --scope <ACCOUNT_RESOURCE_ID>`, then re-run `azd up`. Incremental mode recreates it cleanly under the Bicep GUID.

### Gotcha 2 — function deploy + seed blocked by private-only storage

The function's package/documents storage account is hardened to `publicNetworkAccess=Disabled` (reachable only via private endpoint). `azd deploy function` uploads the package zip as a blob **from the operator's workstation over the public internet**, so the upload is refused with `403 InaccessibleStorageException / BlobUploadFailedException`. RBAC is not the gap (the operator already holds `Storage Blob Data Contributor` + `Owner` on the account). The backend (ACR image push) and frontend (App Service SCM) don't touch this storage, which is why only `function` failed.

- Workaround used: temporarily `az storage account update -n st<SUFFIX> -g <RESOURCE_GROUP> --public-network-access Enabled --default-action Allow`, re-run `azd deploy function`, run the seed, then re-lock with `--public-network-access Disabled`. The function runtime reaches storage via its private path, so re-locking does not break ingestion.
- Same restriction blocks the post-deploy sample-data seed from a workstation — open storage before seeding, re-lock after.
- Secondary finding: the `postdeploy` sample-data hook prints its interactive menu but `input()` hits EOF under azd (no TTY), so the menu choice cannot be read and the seed is skipped (swallowed by `continueOnError: true`). Run the seed non-interactively instead: `AZURE_ENV_SAMPLE_DATA=all` (or `--set all`).

### Gotcha 3 — frontend 503 on first open (BUG-0085)

Opening the frontend returned `503` (Application Error). The App Service container exited on startup with **code 127** (`uvicorn: command not found`) → `ContainerTimeout`. Root cause: the build-from-source frontend (BUG-0081 migration: `PYTHON|3.11` + `uvicorn frontend_app:app` + a staged `requirements.txt`) was missing `SCM_DO_BUILD_DURING_DEPLOYMENT=true` in its `appSettings`, so App Service never ran the Oryx `pip install` and `uvicorn` was absent. Fixed by adding the flag to [main.bicep](../../infra/main.bicep) (recompiled main.json) plus a live `az webapp config appsettings set` + `azd deploy frontend`; the rebuilt container logged `Site started` and the URL now returns `HTTP 302` to the Entra login. Recorded as BUG-0085.

### Gotcha 4 — backend down: reused-resource firewall drift (BUG-0086)

The backend Container App was `ActivationFailed` (0 replicas, every request timed out). Startup logs showed the FastAPI lifespan's `get_runtime_config()` → Cosmos read rejected by the Cosmos firewall (`Forbidden … blocked by your Cosmos DB account firewall settings`) → `Application startup failed. Exiting`. Root cause: this env **reuses pre-existing data resources** (`AZURE_ENV_EXISTING_COSMOS_NAME` / `_STORAGE_NAME` / …); the Bicep data modules are guarded by `!useExisting*`, so they never manage a reused account's network posture. The reused Cosmos (and Storage) were left `publicNetworkAccess=Disabled` from a prior life, but the Container Apps env is **not** VNet-integrated (`enablePrivateNetworking=false`) — so the backend egresses public and Cosmos blocks it. Search + OpenAI happened to be `Enabled`, so only Cosmos (backend) and Storage (function ingestion) broke. Fixed Cosmos with `az cosmosdb update --public-network-access ENABLED --ip-range-filter 0.0.0.0` ("Accept connections from within public Azure datacenters") + a backend revision restart; `/api/health` now returns `200 status:pass` with database/search/foundry_iq all green. Recorded as BUG-0086.

Storage had the same drift; fixed it too (`az storage account update --public-network-access Enabled --default-action Allow --bypass AzureServices`). The `doc-processing` queue holds 6 pending blob-created ingestion events (poison queue empty), but the **function host stays `503`** even after the storage fix + an `az functionapp restart` — the Flex Consumption host won't start, and it emits no App Insights traces (fails before telemetry). The network fix is necessary but not sufficient; the host needs its deployment package re-staged against the now-reachable storage via `azd deploy function`. Backend chat is unaffected (grounds on the existing 289-chunk index); only ingestion of the 2 new docs is blocked.

## Root-cause remediation plan (Option A — reconcile, no data loss)

The recurring cloud breakage collapses to **two** root causes, not a string of unrelated incidents: (1) **reused data resources carry prior-life config the v2 Bicep doesn't manage** (Cosmos/Storage firewalls in BUG-0086, the legacy Event Grid subscription in BUG-0087), and (2) **observability is effectively off**, so each failure is a blind dig. Investigating (2) uncovered BUG-0089: the backend reads `environment=local` despite `AZURE_ENVIRONMENT=production` on the running revision — which both opens the admin auth bypass (security) **and** suppresses App Insights telemetry (the reason BUG-0088 captured nothing). Operator chose **Option A**: keep reusing the data resources and make drift self-healing in `post_provision.py`; no data loss, the existing 393-chunk index is preserved.

Units execute one-at-a-time, test-first, in this order:

- **U1 — BUG-0089 backend settings-read defect (top priority; security + unblocks observability).** Root-cause why the in-process `AppSettings.environment` / `ObservabilitySettings.app_insights_connection_string` read defaults while the container env has the correct values (inspect the backend image for a bundled `.env`, confirm image commit vs `settings.py`, verify pydantic-settings source precedence). Fix test-first. Done when an unauthenticated `GET /api/admin/status` is rejected and the status reports `environment:production`, `app_insights_enabled:true`, `auth_enforced:true`.
- **U2 — reused-resource network posture reconcile (`post_provision.py`).** Idempotent `_reconcile_reused_resource_posture()` (sibling of `_ensure_kb_mcp_connection`): when `enablePrivateNetworking=false` **and** a resource is reused (`AZURE_ENV_EXISTING_*` set), ensure Cosmos + Storage `publicNetworkAccess=Enabled` with the Azure-datacenter rule / `AzureServices` bypass the non-VNet profile needs. No-op for VNet or freshly-provisioned resources. Test-first with a mocked control-plane client.
- **U3 — stale Event Grid subscription prune (`post_provision.py`).** Idempotent prune of subscriptions on the reused storage system topic that deliver raw events to the consumer queue (`doc-processing`) with plain (non-MI) delivery — removes the BUG-0087 drift class, never the v2 `cwyd2-blob-created-*` MI subscription or the Defender antimalware subscription. Test-first with a mocked Event Grid client.
- **U4 — BUG-0088 `.docx` ingestion root cause + fix.** With U1's telemetry live, re-trigger the failing `.docx` (or download the blob + run the v2 parser under pytest) to capture the real exception, then fix test-first.
- **U5 — Backlog C bulk reprocess (operational, after U4).** `POST /api/admin/documents/reprocess` to re-index the ~90 blobs whose Event Grid events expired during the outage. Cost-bearing — operator-triggered, confirm scope first.

**Cross-cutting deploy discipline:** `azd deploy <service>` updates only the container image; env/infra changes reach the running app only via `azd provision` (or `azd up`). Standardize on `azd up` for any change that touches Bicep, to stop the deployed-state-vs-IaC confusion that cost time this session.

## U1 — BUG-0089 root cause (deployment-state, not code)

Investigated 2026-06-25. **The settings code is correct** — a local repro from a clean process env (`AZURE_ENVIRONMENT=production` + App Insights conn string, run from a dir with no `.env`) returns `environment=Environment.PRODUCTION` and `app_insights_enabled=True`. The deployed backend image (built today 00:20Z) contains that code (the `AZURE_ENVIRONMENT` read landed 2026-05-20). So BUG-0089 is **not** a code defect.

**Real cause — Container Apps revision anomaly.** The app is in **Single**-revision mode yet had **two** active revisions:

| Revision | Image | `AZURE_ENVIRONMENT` | Health | Replicas | Traffic |
|---|---|---|---|---|---|
| `0000012` (Jun) | old | (none) | — | 1 running | 0 |
| `azd-1782433210` (today) | current | `production` | Healthy/Provisioned | 0 (Stopped) | 100 |

The current revision (production env) is healthy but scaled to zero; the stale `0000012` (created before the 2026-06-22 Bicep `AZURE_ENVIRONMENT` wiring, so **no** env var) kept the only running replica and served all traffic as the de-facto fallback → `local` default → admin bypass + telemetry off. Likely trigger: the current revision was created before the BUG-0086 Cosmos-firewall fix, so it crash-looped on the startup Cosmos call and the platform fell back to the last-healthy old replica.

**Incident (self-inflicted, resolved).** Deactivating `0000012` to force the cutover took the backend **down** (the current revision didn't cold-start into the gap → 404); a restart didn't bring it up in the window. **Reactivated `0000012` → service restored** (`/api/health` 200). Net state is unchanged from before the turn (old revision serving, bypass still on).

**Fix (pending, controlled cutover):** make the current revision the sole serving revision now that Cosmos/Storage are reachable — either `azd deploy backend` (fresh single-revision cutover) or warm the current revision (`minReplicas≥1`), verify it reports `environment=production`, then deactivate `0000012`. Do **not** deactivate the old revision while the current is scaled-to-zero.

**Resolved (Option B).** `azd deploy backend` (run from `v2/`) created a fresh revision `azd-1782445947` that starts cleanly now that Cosmos is reachable — it holds a **Running** replica and is the **sole active revision** (clean Single-mode cutover; `0000012` + `azd-1782433210` deactivated). Verified: `/api/health` → `auth_enforced=true`; unauthenticated `GET /api/admin/status` → **401** (bypass closed). The admin auth gate enforces again; telemetry is wired (same env load that yields `production`). Postdeploy sample-data seed was **skipped** (`0`). **BUG-0089 closed.**

## BUG-0090 — production admin unreachable: Easy Auth identity source never wired

Found 2026-06-25, immediately after BUG-0089 closed. Operator hit `GET /api/admin/status` on the backend FQDN and got `401 {"detail":"Missing client principal claims; Easy Auth claims header required."}`.

**This 401 is the gate working as designed, not a regression.** In `production`, `requires_role("admin")` ([dependencies.py](../../src/backend/dependencies.py)) requires the base64 `x-ms-client-principal` Easy Auth claims blob carrying an `admin` role. Closing the BUG-0089 bypass exposed a pre-existing gap.

**Root cause — no Easy Auth identity source feeds the backend.** Confirmed live:

- Backend Container App `ca-backend-<SUFFIX>`: `az containerapp auth show` → `{}`. **No Easy Auth.** No claims blob is ever injected.
- `frontend_app.py` serves the SPA + `GET /config` only — it does **not** proxy `/api/*`. The SPA calls the backend **directly** cross-origin and cannot forge the trusted `x-ms-client-principal` header.
- No Entra app registration defining an `admin` app role exists in the deployment.
- Frontend App Service `app-frontend-<SUFFIX>` has Easy Auth enabled but its Entra provider reads back **unconfigured** (empty `clientId`/issuer) — the identity layer was never fully provisioned.

Admin only *appeared* to work before via the local-dev bypass on the stale revision (the BUG-0089 anomaly). Non-admin chat/history still works because the SPA forwards a default `x-ms-client-principal-id` that `get_user_id` accepts in production.

**Fix is an auth-architecture decision (Hard Rule #10 — needs operator sign-off):**

- **Option A — backend Easy Auth.** Enable Container Apps Easy Auth on the backend behind an Entra app registration with an `admin` app role; assign the operator. SPA drives the backend `/.auth` login (cross-origin → needs CORS + credentialed fetch). Real multi-user RBAC; most work; partly manual Entra steps; `azd provision`.
- **Option B — frontend reverse-proxy.** Make `frontend_app.py` proxy `/api/*` to the backend and forward the Easy Auth headers the frontend injects; define the `admin` role on the frontend app registration. Single origin, one sign-in, no CORS. Structural change to `frontend_app.py` + infra; the frontend's Easy Auth provider must first be fully configured.

Recorded as BUG-0090 (open — awaiting decision). **Not implemented this turn** — structural + security decision for the operator.
