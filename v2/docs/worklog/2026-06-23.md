## Summary

Fixed the recurring `azd up` provision failure where the Event Grid event subscription's managed identity is not authorized to deliver to the `doc-processing`/`blob-events` Storage Queue (aka.ms/egmsivalidation). This is the durable bicep back-port of BUG-0061.

## Planned

* Stop the deterministic Event Grid MI-delivery 401 on fresh `azd up`.

## Done

* [v2/infra/main.bicep](../../infra/main.bicep): restructured the new-topic Event Grid wiring so the role assignment precedes the subscription's synchronous MI-delivery preflight.
  * The AVM `event-grid/system-topic:0.6.4` module now declares `eventSubscriptions: []` (no embedded subscription).
  * Added a standalone `newEventGridSubscription` (`Microsoft.EventGrid/systemTopics/eventSubscriptions`) parented to a new `existing` reference `newEventGridTopic`, that `dependsOn: [ eventGridQueueSenderRole ]`.
  * This mirrors the pre-existing `useExistingEventGridTopic` branch (`existingEventGridSubscription` → `existingQueueMessageSenderRole`).
* `az bicep build` clean (only the pre-existing BCP334 container-registry-name warning); `main.json` regenerated.

## Bugs

* [BUG-0061](../bugs.md) — durable bicep back-port applied (was previously fixed only via an out-of-band live role grant). Closed pending follow.

## Decisions

* Took option (a) from the BUG-0061 durable-fix note (split the subscription out of the module) over option (b) (two-pass provision): it reuses the standalone-subscription pattern already present in the file for the existing-topic branch, so it is the lowest-risk structural change.
