### Step 2 BLOCKED — Flex remote-build host wedged (BUG-0058 hard manifestation)

> **Correction (2026-06-24) — the conclusion in this section is WRONG.** This
> was **not** a platform Flex build-service outage. The remote Oryx build wedged
> because `pip install` could not resolve `hyperlight-sandbox-backend-wasm` on
> the host's **Python 3.11**, pulled transitively by the **umbrella**
> `agent-framework==1.7.0` dependency (its `agent-framework-hyperlight`
> requirement is gated `python_version < "3.14"`, so local Python 3.14 never
> reproduced it). Repinning to `agent-framework-core==1.7.0` and deploying the
> pre-built package with `--no-build` — which the analysis below wrongly
> predicted "would not help" — **succeeded** on 2026-06-24 (`blob_event` is now
> live). See **BUG-0080** and [worklog/2026-06-24.md](2026-06-24.md). The notes
> below are retained as a record of the misdiagnosis.

Step 1 (prepackage) succeeded — `build-functions/` staged with `blob_event`
(0.6 MB source-only package, 126 files; intended Flex **remote build**).

Step 2 (`azd deploy function`) failed repeatedly. Diagnosis from
`deployment-package/kudu-state.json` (the Flex remote-build state blob):

* `released-package.zip` (live package) still dated **2026-06-17**
  (`ActiveDeploymentId = f530ad19`, last healthy build — Oryx took ~85 s).
* First attempt `d42c092c` (18:21): registered, then `Kudu-OryxBuildStep`
  with **StartTime 18:21:30 and no EndTime for ~47 min** → hung. A plain
  `az functionapp restart` only *resumed* the hung build and it held the
  deployment lock → every new deploy fast-failed with **"deployment was
  cancelled due to another deployment being in progress."**
* `az functionapp stop` → `start` cleared the lock (forceful build-host kill).
* Retry `b4b29a5a` (19:13): registered but sat at **`status=0`, zero build
  steps, no Oryx** for 9+ min → wedged again. `released-package.zip` never
  republished; `blob_event` never appeared.

Ruled out: **not** our package (clean 0.6 MB source), **not** VNet egress
(`vnetSubnet = null`, no route-all), **not** storage auth (today's
`StorageAccessCheck.txt` passed), **not** a custom build setting (none set).
Two consecutive fresh deployments wedge at the build host with zero steps →
**platform-side Flex remote-build-host failure** for this app/region.

Remediations attempted (operator-approved, in order):

* **(A) Cooldown + retry** — stop/start to recycle the build host, ~8 min
  cooldown, retry. Result: deploy `aa3a8f67` wedged identically (zero steps).
* **(1) Clear the build ledger + retry** — backed up then deleted
  `deployment-package/kudu-state.json`, recycled, retried. The running app was
  **unaffected** (5 functions kept serving — confirms the runtime mounts
  `released-package.zip`, not the ledger). First retry fast-failed ("another
  deployment in progress" — the lock lives in the **control plane**, not the
  blob); a clean stop/start cleared it; retry `2968adc9` wedged again with the
  **fresh** ledger → disproves the stuck-records hypothesis.

**Refined diagnosis (4 consecutive wedges: `d42c092c`, `b4b29a5a`, `aa3a8f67`,
`2968adc9`):** every deploy registers then executes **zero build steps** — the
wedge happens **before even `Kudu-SourcePackageUriDownloadStep`**, i.e. the Flex
**build worker is not picking up any job at all** (not merely failing in Oryx).
This is independent of ledger state, cooldown, and recycle. With the 2026-06-20
failure (`331af870`: "Kudu has been restarted during deployment"), the evidence
points to a **platform-side Flex remote-build-service failure** for this app
(UK South) that began ~2026-06-20. Resource Health is N/A for Consumption/Flex
plans, so no portal corroboration. Ruled out earlier: our package (clean 0.6 MB),
VNet egress (none), storage auth (today's check passed), build settings (none).

Because the build worker ignores *all* jobs (not just remote builds), a
"no-remote-build" (pre-built package) deploy via the same OneDeploy path would
likely wedge the same way — Option B is **not** a confident fix while the worker
is down. Realistic paths now: (3) wait hours for the platform build service to
recover, then retry; (B) attempt a no-build deploy as a long shot; (4) pause the
cutover and work other bugs locally meanwhile. Cutover Steps 3–7 remain blocked
on getting `blob_event` deployed. Backup of the original ledger saved to
`%TEMP%\kudu-state-backup.json`.

**Confirmation via a second tool (`func azure functionapp publish`) + an
uninterrupted window:** switched from azd to Core Tools v4.12.0 to get a
different upload path and streamed build logs. Finding: **the upload works
fine** (`func` uploaded 236 KB in seconds — so azd's endless "Uploading
deployment package" label was misleading; the upload is not the bottleneck).
`func` then secured the build slot (deployment `88ea52ec`) with no "another in
progress" cancellation — so the deployment lock is not the blocker either. The
build was then left **completely untouched for ~17 minutes** (20:16 → 20:33,
zero stop/starts) and executed **zero build steps** — never reached even
`Kudu-SourcePackageUriDownloadStep`, package never republished. This refutes the
"builds were being aborted too early by my stop/starts" theory: even fully
uninterrupted, the build pipeline does not execute. Net: across **4 azd attempts
+ 2 func attempts + 1 uninterrupted 17-min window**, the Flex remote-build
pipeline never runs a single step for this app. Upload OK, build pipeline dead →
**platform-side Flex build-service failure (UK South)**, not fixable client-side.
A no-remote-build (pre-built) deploy routes through the same dead OneDeploy
pipeline (which stalls before the source-download step), so it would not help

other open bugs locally meanwhile.

## BUG-0055 investigation (App Insights zero telemetry) — root cause found

Pivoted to BUG-0055 while the cloud deploy is blocked. **Backend half root cause
confirmed (a name mismatch):**

* Bicep wires **all three** runtimes (backend ACA, frontend, function) with the
  env var **`APPLICATIONINSIGHTS_CONNECTION_STRING`** (`main.bicep` L1831 / L1945
  / L2122 → `applicationInsights.outputs.connectionString`). The
  `AZURE_APP_INSIGHTS_CONNECTION_STRING` at L2473 is only an **azd output**, not
  a runtime container var.
* The backend code reads **only** `AZURE_APP_INSIGHTS_CONNECTION_STRING`:
  `ObservabilitySettings` (`backend/core/settings.py`) has
  `env_prefix="AZURE_"` + field `app_insights_connection_string`. `app.py`
  lifespan calls `configure_azure_monitor(connection_string=...)` only when that
  typed field is non-empty.
* Local `.env` uses `AZURE_APP_INSIGHTS_CONNECTION_STRING` (so local telemetry
  works); the cloud container only has `APPLICATIONINSIGHTS_CONNECTION_STRING`,
  which the code ignores → `configure_azure_monitor` never fires → **zero
  backend telemetry in the cloud**.

**Deliberate-decision conflict (Hard Rule #0):** two existing tests in
`tests/backend/test_app_lifespan.py` *intentionally* enforce reading only the
typed name and *ignoring* `APPLICATIONINSIGHTS_CONNECTION_STRING`
(`test_lifespan_configures_app_insights_from_typed_settings` cites CU-002b;
`test_lifespan_skips_app_insights_when_typed_setting_empty` cites CU-007). So the
fix touches a tested decision — surfaced to the operator rather than reversed
unilaterally. Fork: (A) align Bicep to set `AZURE_APP_INSIGHTS_CONNECTION_STRING`
on the backend container (preserves the code decision; bicep edits paused; live
`az containerapp update` can apply it now); (B) make the settings field also
accept the standard `APPLICATIONINSIGHTS_CONNECTION_STRING` via `AliasChoices`
(matches Azure-wide convention + what `configure_azure_monitor` reads natively,
but reverses CU-002b/CU-007 and updates their 2 tests).

**Function half (worker OTel wired, 2026-06-23):** added `_configure_telemetry()`
to `src/functions/function_app.py` (module-load) — reads the host-provided
`APPLICATIONINSIGHTS_CONNECTION_STRING` and calls `configure_azure_monitor(...)`
so the Python worker's application telemetry exports to App Insights; no-ops when
the var is absent. Test-first: two `tests/functions/test_function_app.py` tests
(configured + no-op). Host-level `requests` telemetry is host-emitted (not the
worker), so enabling `host.json` `telemetryMode` is a possible follow-up if those
stay absent after deploy. Gates green: function_app tests 7 passed, all 998
`tests/shared` source-discipline gates passed, no type/lint errors. Not yet
cloud-verified (deploy blocked).

## BUG-0058 fix (prepackage hook now service-scoped)

Root-caused the open question via the azd schema (`azure.yaml.json`): a
**project-level** `hooks.prepackage` "Runs before the `package` command" only,
so a targeted `azd deploy function` (packages just that service) never fired it
— shipping a stale `build-functions/`. A **service-level** `prepackage` hook
"Runs before the service is deployment package is created", which includes
`azd deploy <service>`. Fix: moved `prepackage` from project-scoped
`hooks.prepackage` to `services.function.hooks.prepackage`; run path is now
service-relative (`../scripts/prepackage-function.{sh,ps1}`), and the Python
script is already `__file__`-relative so cwd is irrelevant. Added a drift-guard
[tests/functions/test_azure_yaml_hooks.py](../tests/functions/test_azure_yaml_hooks.py)
(hook is service-scoped + no project-level prepackage returns). Gates green: 293
`tests/functions` pass, prepackage script re-staged all six subpackages incl.
`blob_event`. Stays **open** only for the live `azd deploy function`
confirmation (same blocked deploy). The manual prepackage workaround is no longer
needed once this ships.
