## Done

* **Unit 2a ‚Äî by-key read primitive (test-first).** Added `async def get_document_by_key(self, key: str) -> SearchResult | None` to [base.py](../../src/backend/core/providers/search/base.py) (non-abstract default that raises `NotImplementedError` naming the concrete class, so an un-overriding provider fails loud at the call site rather than silently skipping enrichment ‚Äî same contract as `list_sources`) and the concrete override to [azure_search.py](../../src/backend/core/providers/search/azure_search.py). The override calls the SDK `client.get_document(key=key, selected_fields=["id", "content", "title", "url"])`, maps the row through the existing `_to_result`, and owns the Hard Rule #14 boundary: a missing key (`ResourceNotFoundError`) returns `None` (a soft miss ‚Äî a re-ingested or deleted key degrades enrichment to the raw id, never crashes the answer), while any other `AzureError` is logged with the canonical structured `extra` (`operation="get_document_by_key"`, `provider="azure_search"`, `index_name=‚Ä¶`) and re-raised. `ResourceNotFoundError` is caught **before** `AzureError` (it is a subclass). A by-key fetch carries no relevance score, so the resulting `SearchResult.score` is `None`.
* **Why a provider method, not an inline SDK call in the orchestrator.** Keeps the Azure SDK `try/except` + resilience on the search provider (where every other Search SDK boundary lives) and keeps `citations.py` provider-free ‚Äî the unit-1 `fetch_document` seam is satisfied by a thin closure over `search.get_document_by_key` in the wire-in, with no new SDK import on the citation-format path (ADR 0026 / Hard Rule #20 R2).
* **Tests.** Base default `NotImplementedError` (+ not-abstract guard) in [test_base.py](../../tests/backend/core/providers/search/test_base.py); concrete happy-path field mapping, `key` + `selected_fields` argument assertions, not-found ‚Üí `None` (asserts nothing logged), and `AzureError` ‚Üí log + re-raise in [test_azure_search.py](../../tests/backend/core/providers/search/test_azure_search.py).

### What shipped (one unit + consuming edits)

* **Dependency.** Added `remark-supersub@^1.0.0` to `src/frontend/package.json` (the same version v1 pairs with `react-markdown@^10`) and ran `npm install` at the v2 workspaces root. The package ships its own types (`lib/index.d.ts`) and a modern `"exports"` entry, so **no ambient `.d.ts` shim was needed** ‚Äî Vite, Vitest, and strict `tsc` all resolve it.
* **Unit: `parseAnswer.tsx`** (Pillar: Stable Core; Phase 6 visual polish). * **`MarkdownContent` `enableSupersub` prop.** Opt-in; when set, adds `remark-supersub` to the remark pipeline so ` ^K^ ` renders as `<sup>`. Still no `rehype-raw`, so raw HTML stays escaped (XSS-safe). The reasoning panel leaves it off so stray `^`/`~` in chain-of-thought stays literal.
* **`MessageList` wiring (W2).** The assistant answer body now renders `parseAnswer(m.content, m.citations).markdownText` with `enableSupersub`. The `CitationPanel` deliberately stays on the unmodified `m.citations` this turn (no regression if an answer has no inline markers); rewiring the panel to the renumbered subset is BUG-0015. User messages and citation-less answers keep `[docN]` verbatim.

### Cloud architecture note ‚Äî the link needs `VITE_BACKEND_URL`

The link must therefore use the absolute `${VITE_BACKEND_URL}/api/files/<filename>` ‚Äî the same pattern `App.tsx` and `HistoryPanel.tsx` already use. ## Detour ‚Äî chat layout / UX fix (operator-reported, plan-mode review first)

### Root cause ‚Äî Problem A (double scroll / floating composer / header off-screen)

The DOM chain from `#root` down is `#root` (height:100%) ‚Üí `div.fui-FluentProvider` (the wrapper Fluent v9's `<FluentProvider>` renders in [FluentThemeBridge.tsx](../../src/frontend/src/theme/FluentThemeBridge.tsx)) ‚Üí `.column` (CoralShell). The Fluent wrapper had **no height rule**, so it defaulted to `height:auto`; to compensate, `.column` used `height:100vh` instead of `height:100%`. `100vh` includes the strip under a horizontal scrollbar, so when a wide markdown table overflows the message column on the X axis the resulting scrollbar makes `100vh` taller than the visible viewport and the **page** (body) scrolls ‚Äî sliding the header (which sits in `.column` above the scroll row) off the top and lifting the composer off the bottom, exposing the whitespace below. v1 never hits this because it chains `height:100%` end-to-end and never uses `100vh` on the shell.

### Done ‚Äî L1 (repair the shell height chain, CSS + 1 test)

* **The Fluent wrapper now carries the height chain, and the shell uses `height:100%` (not `100vh`).** Tagged `<FluentProvider>` with a stable `className="appFluentRoot"` ([FluentThemeBridge.tsx](../../src/frontend/src/theme/FluentThemeBridge.tsx)) and added `.appFluentRoot { height: 100% }` to the global [tokens.css](../../src/frontend/src/theme/tokens.css) (with a comment explaining why the wrapper needs it). ## BUG-0046 ‚Äî frontend user identity + per-user `x-ms-client-principal-id` forwarding (plan + Phase 0 tracking)

The operator asked to give the v2 frontend a real per-user identity and forward it to the backend on every API call, adapting the v1 `getUserInfo()` flow and the MACAE header-forward pattern (not MSAL / JWT): on load, detect whether login is enforced; if so, resolve the signed-in user and make it app-wide available, else use the default user `00000000-0000-0000-0000-000000000000`; show an error screen when login is enforced but absent; pass `x-ms-client-principal-id` on every API call; and on the backend confirm the id is present and well-formed. Filed as **BUG-0046** (frontend, high, open) in [bugs.md](../bugs.md); the running task list is [frontend-user-identity-plan.md](../frontend-user-identity-plan.md).

### Key finding ‚Äî the backend is already user-scoped

Research (two read-only Explore passes + backend reads) confirmed the "query history by user" and "a deleted chat must belong to the same user" requirements are already satisfied server-side: the history routes (`list` / `get` / `delete` / `rename` / `add_message` / `feedback`) and the conversation route consume `UserIdDep` (which reads `x-ms-client-principal-id`); `get_user_id` returns the header value when present, falling back to `local-dev` in `local` mode and `401` in `production` mode; and the Cosmos DB / PostgreSQL providers key every read and write on `user_id` (a non-owned conversation `404`s). The defect is entirely in the frontend, which never resolves the user and forwards no identity header on any of its four `fetch` clients (`streamChat`, `conversationHistory`, `admin`, `speech`). The separate-origin topology (SPA ‚Üí absolute `VITE_BACKEND_URL`) means the platform-injected Easy Auth header never reaches the cross-origin backend call, so the identity must be forwarded manually (MACAE) ‚Äî and per-user isolation is non-functional until it is.

### Security constraint

A browser-forwarded `x-ms-client-principal-id` is not a trust boundary (any caller can forge it). It is acceptable for chat-history partitioning, but admin RBAC stays anchored on the backend's own server-injected Easy Auth claims (`x-ms-client-principal`); the browser must not forward the claims blob, and a "valid" forwarded id means well-formed, not authentic.

### Plan (one unit per turn, test-first)

* **Phase 1 ‚Äî backend:** B1 fold `auth_enforced` into `GET /api/health` (`= settings.environment is Environment.PRODUCTION`); B2 (optional) light principal-id well-formedness check in `get_user_id`.
* **Phase 2 ‚Äî `api/auth.tsx`:** F1 auth models + `DEFAULT_USER_ID`; F2 `getUserInfo()` over `/.auth/me`; F3 `getUserId` / `setUserId` / `userIdHeaders()`.
* **Phase 3 ‚Äî header forwarding:** F4‚ÄìF7 spread `userIdHeaders()` into each of the four clients (one file per unit).
* **Phase 4 ‚Äî bootstrap + UI:** F8 auth context / state; F9 bootstrap effect in `AppShell`; F10 blocked-screen component (Fluent v9 port of the v1 "Authentication Not Configured" screen); F11 render it when enforced but no user.
* **Phase 5 ‚Äî verify:** FE vitest + `tsc -b` (both trees) + eslint; BE pytest + `pyright --strict` + shared gates; live two-user isolation check.
