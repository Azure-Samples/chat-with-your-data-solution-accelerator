## Done

* **Unit 2a — by-key read primitive (test-first).** Added `async def get_document_by_key(self, key: str) -> SearchResult | None` to [base.py](../../src/backend/core/providers/search/base.py) (non-abstract default that raises `NotImplementedError` naming the concrete class, so an un-overriding provider fails loud at the call site rather than silently skipping enrichment — same contract as `list_sources`) and the concrete override to [azure_search.py](../../src/backend/core/providers/search/azure_search.py). The override calls the SDK `client.get_document(key=key, selected_fields=["id", "content", "title", "url"])`, maps the row through the existing `_to_result`, and owns the Hard Rule #14 boundary: a missing key (`ResourceNotFoundError`) returns `None` (a soft miss — a re-ingested or deleted key degrades enrichment to the raw id, never crashes the answer), while any other `AzureError` is logged with the canonical structured `extra` (`operation="get_document_by_key"`, `provider="azure_search"`, `index_name=…`) and re-raised. `ResourceNotFoundError` is caught **before** `AzureError` (it is a subclass). A by-key fetch carries no relevance score, so the resulting `SearchResult.score` is `None`.
* **Why a provider method, not an inline SDK call in the orchestrator.** Keeps the Azure SDK `try/except` + resilience on the search provider (where every other Search SDK boundary lives) and keeps `citations.py` provider-free — the unit-1 `fetch_document` seam is satisfied by a thin closure over `search.get_document_by_key` in the wire-in, with no new SDK import on the citation-format path (ADR 0026 / Hard Rule #20 R2).
* **Tests.** Base default `NotImplementedError` (+ not-abstract guard) in [test_base.py](../../tests/backend/core/providers/search/test_base.py); concrete happy-path field mapping, `key` + `selected_fields` argument assertions, not-found → `None` (asserts nothing logged), and `AzureError` → log + re-raise in [test_azure_search.py](../../tests/backend/core/providers/search/test_azure_search.py).
* **Gates.** `tests/backend/core/providers/search/test_azure_search.py` + `test_base.py` **49 passed**; `pyright --strict` **0 / 0 / 0** on both `azure_search.py` and `base.py`; `get_errors` clean on all four touched files.

## Implementation — BUG-0030 fallback (b), unit 2b (orchestrator wire-in, 2026-06-15)

### Done

* **Captured the injected search provider.** `AgentFrameworkOrchestrator.__init__` gained a keyword-only `search: BaseSearch | None = None` parameter (stored as `self._search`); previously the router's uniform `search=…` kwarg was absorbed by `**_extras`. The `**_extras` comment was corrected to drop `search` from the swallowed list. No closure was needed after all — `enrich_kb_citations` takes the callable directly, so the wire-in passes the bound method `search.get_document_by_key`, which already matches the `Callable[[str], Awaitable[SearchResult | None]]` seam.
* **Best-effort enrichment in `run()`.** Right after `normalize_kb_citations`, when a search provider is wired, `run()` calls `await enrich_kb_citations(citations, search.get_document_by_key)`. langgraph-shaped citations and unresolved keys pass through unchanged (handled inside `enrich_kb_citations`). The call is wrapped in `try/except AzureError` → `logger.warning(..., extra={"operation": "enrich_citations", "provider": "agent_framework", "agent_name": …})`: enrichment is cosmetic and the answer is already fully assembled, so a transient by-key lookup failure degrades to the normalized raw-id citations rather than failing the whole answer. The underlying SDK failure was already logged with a traceback at the `get_document_by_key` boundary (unit 2a), so the warning here avoids a duplicate stack trace while still satisfying the silent-except gate.
* **Local-variable collision fix.** `__init__` reused a local named `search` (`search = settings.search`, a `SearchSettings`) to capture the infra-pinned KB scalars. The new `search` parameter (`BaseSearch | None`) collided with it, producing 28 cascading `pyright --strict` errors (the local reassignment retyped `search`, so every `self._search_endpoint` / `self._kb_name` derived from it became `Unknown`). Renamed the local to `search_settings` — a minimal, self-contained rename that also reads more clearly (config object vs. provider).
* **Tests (test-first).** Added to [test_agent_framework.py](../../tests/backend/core/providers/orchestrators/test_agent_framework.py): `test_run_enriches_kb_citations_via_search_lookup` (a KB citation keyed `mcp://searchindex/abc123` → the bare key `abc123` drives `get_document_by_key`, and the resolved document backfills `title` / `snippet` / `url` while the id stays the normalized `[doc1]` and `metadata.source_id` keeps the raw id for traceability) and `test_run_keeps_raw_citations_when_enrichment_lookup_fails` (an `HttpResponseError` from the lookup degrades to the raw-id citation — answer + citation still stream, and the `enrich_citations` warning is logged, not swallowed). Threaded a `search` parameter through `_make_orchestrator`, imported `SearchResult`, added an `_AGENT_FW_LOGGER_NAME` constant, and corrected the `test_constructor_swallows_uniform_extras` docstring (`search` is now a named parameter, consumed not swallowed).

### Live capture 1 — title/snippet confirmed, url defect found

The served citations now carry the **friendly** fields the enrichment backfills: `id="[doc1]"`, `title="Benefit_Options.pdf"`, `snippet=<chunk text>`, no native `【N:M†source】` markers, and `metadata["source_id"]` retaining the raw `mcp://searchindex/<key>`. One defect surfaced in the same capture: the served `url` was the **literal string `"None"`**. Root cause in [azure_search.py](../../src/backend/core/providers/search/azure_search.py) `_to_result`: `url=str(doc.get("url", ""))` — this sample document has a present-but-`null` `url` field, so `dict.get("url", "")` returns `None` (the `""` default only fires when the key is *absent*), and `str(None)` → `"None"`. The enrichment newly exercised this path (it backfills `citation.url` from the by-key document), and it is a **second** `str(None)`→`"None"` site, distinct from BUG-0033's `citations_from_annotations` annotation-boundary fix. The operator chose to fix it now (test-first) as the last piece of BUG-0030 rather than file it separately.

### Null-url fix (test-first)

* **Test first.** Added `test_get_document_by_key_maps_null_url_to_empty_string` to [test_azure_search.py](../../tests/backend/core/providers/search/test_azure_search.py): a by-key document whose `url` field is `None` must map to `SearchResult.url == ""`, never `"None"`. It failed against the current code (`assert 'None' == ''`), confirming the bug.
* **Fix.** Changed `_to_result` `url=str(doc.get("url", ""))` → `url=str(doc.get("url") or "")` (a present-but-`null` or empty `url` now coerces to `""`). One line; strictly an improvement; also cleans the `langgraph` retrieval path for url-less documents.

## BUG-0016 — inline `[docN]` markers render as v1-style superscripts

Picked up the FE citation cluster after BUG-0030 closed. Scoped BUG-0016 first (inline marker rendering); BUG-0015 (references block + right panel) follows.

Confirmed two decisions with the operator before touching layout:

### What shipped (one unit + consuming edits)

* **Dependency.** Added `remark-supersub@^1.0.0` to `src/frontend/package.json` (the same version v1 pairs with `react-markdown@^10`) and ran `npm install` at the v2 workspaces root. The package ships its own types (`lib/index.d.ts`) and a modern `"exports"` entry, so **no ambient `.d.ts` shim was needed** — Vite, Vitest, and strict `tsc` all resolve it.
* **Unit: `parseAnswer.tsx`** (Pillar: Stable Core; Phase 6 visual polish). Pure transform: rewrites each in-bounds `[docN]` marker into a ` ^K^ ` superscript token, dedupes repeated sources by `citation.id`, renumbers them `1..K` in first-appearance order, collapses consecutive duplicate superscripts, and leaves out-of-bounds / `[doc0]` markers verbatim. Returns `{ markdownText, citations }` where `citations` is the deduped/renumbered referenced subset (that subset feeds BUG-0015's numbered block later).
* **`MarkdownContent` `enableSupersub` prop.** Opt-in; when set, adds `remark-supersub` to the remark pipeline so ` ^K^ ` renders as `<sup>`. Still no `rehype-raw`, so raw HTML stays escaped (XSS-safe). The reasoning panel leaves it off so stray `^`/`~` in chain-of-thought stays literal.
* **`MessageList` wiring (W2).** The assistant answer body now renders `parseAnswer(m.content, m.citations).markdownText` with `enableSupersub`. The `CitationPanel` deliberately stays on the unmodified `m.citations` this turn (no regression if an answer has no inline markers); rewiring the panel to the renumbered subset is BUG-0015. User messages and citation-less answers keep `[docN]` verbatim.

### Tests

* **`parseAnswer.test.tsx`** — 13 cases: undefined/empty passthrough, no-markers passthrough, single marker, multiple distinct in order, dedupe, out-of-order renumber, out-of-bounds `[doc9]`/`[doc0]` verbatim, consecutive-duplicate collapse, distinct-adjacent not collapsed, referenced-subset, mixed in/out-of-bounds.
