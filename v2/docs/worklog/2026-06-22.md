## Summary

Ran down why the header still showed an **"M"** after the BUG-0071 work. The earlier diagnosis ("stale Vite dev server" → restart) was **wrong** and the restart was an unnecessary action. Read the Fluent v9 Avatar source: [useAvatar.js](../../node_modules/@fluentui/react-avatar/lib/components/Avatar/useAvatar.js) renders the `icon` slot **only when there are no initials** (`if (!initials && (!image || imageHidden)) { icon = ... }`). The header Avatar carried a `name` (`"Microsoft"`, then `"Multi-agent"`), which always yields initial **"M"**, so the `icon` logo was **never** rendered — the original 4-square never showed either. Every glyph swap (`MsftColorLogo` → `PeopleTeam` → `MultiAgentLogo`) was masked by the same `name`-suppresses-`icon` behaviour.

## BUG‑0071 — real fix

* **Diagnosis first (no code changes until proven):** confirmed the dev server *was* running on :5273 (PID 47860), then read the Avatar source to prove the `name`/`icon` precedence. The dev server was never the cause.
* **Fix (one line):** removed the `name="Multi-agent"` prop from the `<Avatar>` in [Header.tsx](../src/frontend/src/components/Header/Header.tsx). With no `name` → no initials → Fluent renders the `icon` slot, so `<MultiAgentLogo size={20} />` (the MACAE interlocking glyph) shows. Mirrors MACAE's header Avatar, which has no `name`.
* **Regression guard:** added a [Header.test.tsx](../tests/frontend/components/Header.test.tsx) case asserting the `MultiAgentLogo` svg (`viewBox="0 0 33 32"`) actually renders — this fails the instant a `name` prop re-suppresses the icon.
* **Verified:** Header **11 passed** (incl. the new test) + MultiAgentLogo **3 passed**, eslint clean. The change is an edit to an existing file, so Vite **HMR applies it without a restart** — refresh the browser.
* **Lesson (also in repo memory):** never pass Fluent `Avatar` a `name` when you want a custom logo to show — `name` initials always win over `icon`.

## BUG‑0075 — documents list "last modified" is blank (pgvector)

Analysis-first: `GET /api/admin/documents` → `list_sources()` returned `last_modified: null` for every source on **both** backends by design — the pgvector `documents` table had **no timestamp column** (insert wrote only `id/content/title/content_vector`; `list_sources` hardcoded `last_modified=None`), and Azure Search's facet aggregation also emits `None`. So nothing ever stored or computed the timestamp.

Fix (Option A — operator-approved; pgvector, the live backend):
* **Schema** — added `last_modified TIMESTAMPTZ NOT NULL DEFAULT now()` to the `documents` CREATE TABLE **plus** an idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS` in `ensure_schema` (the backend lifespan runs it on startup, so deploys whose table predates the column backfill automatically; existing rows take the migration time until re-ingested). [pgvector.py](../src/backend/core/providers/search/pgvector.py).
* **Insert** — `merge_or_upload_documents` lets new rows default to `now()`; on `ON CONFLICT DO UPDATE` it sets `last_modified = now()` so re-ingest refreshes the timestamp.
* **List** — `list_sources` now selects `MAX(last_modified)` per source and emits the ISO‑8601 string (None defensively mapped).
* **Verified live:** `GET /api/admin/documents` now returns populated `last_modified` (existing 3 sources show the migration time `2026‑06‑22T19:39:14Z`). 31 pgvector tests, AST gates **357 passed**, pyright **0/0/0**. Also confirmed BUG‑0074's HTML fix end‑to‑end — the `project‑resolve` URL ingested as `…project-resolve.html` into **200 clean chunks**.
* Azure Search `last_modified` stays `None` (its facet path has no timestamp field) — out of scope for this pgvector deployment; documented on `SourceListing`.

## BUG‑0069 — deployed backend reports `environment: "local"` (admin-auth fail-open risk)

The deployed backend's `GET /api/admin/status` reported `"environment": "local"` because `AppSettings.environment` (env var `AZURE_ENVIRONMENT`, default `Environment.LOCAL`) was **never wired** onto the backend Container App in bicep. Security weight: the admin auth local-dev bypass in [requires_role](../src/backend/dependencies.py) returns the synthetic `local-dev` admin **only** when `environment == 'local'` and the Easy Auth claims blob is absent — so a cloud runtime mis-reporting `local` could grant admin without authentication.
