## Summary

Ran down why the header still showed an **"M"** after the BUG-0071 work. The earlier diagnosis ("stale Vite dev server" → restart) was **wrong** and the restart was an unnecessary action. Read the Fluent v9 Avatar source: [useAvatar.js](../../node_modules/@fluentui/react-avatar/lib/components/Avatar/useAvatar.js) renders the `icon` slot **only when there are no initials** (`if (!initials && (!image || imageHidden)) { icon = ... }`). The header Avatar carried a `name` (`"Microsoft"`, then `"Multi-agent"`), which always yields initial **"M"**, so the `icon` logo was **never** rendered — the original 4-square never showed either. Every glyph swap (`MsftColorLogo` → `PeopleTeam` → `MultiAgentLogo`) was masked by the same `name`-suppresses-`icon` behaviour.

## BUG‑0071 — real fix

* **Diagnosis first (no code changes until proven):** confirmed the dev server *was* running on :5273 (PID 47860), then read the Avatar source to prove the `name`/`icon` precedence. The dev server was never the cause.
* **Fix (one line):** removed the `name="Multi-agent"` prop from the `<Avatar>` in [Header.tsx](../src/frontend/src/components/Header/Header.tsx). With no `name` → no initials → Fluent renders the `icon` slot, so `<MultiAgentLogo size={20} />` (the MACAE interlocking glyph) shows. Mirrors MACAE's header Avatar, which has no `name`.
* **Regression guard:** added a [Header.test.tsx](../tests/frontend/components/Header.test.tsx) case asserting the `MultiAgentLogo` svg (`viewBox="0 0 33 32"`) actually renders — this fails the instant a `name` prop re-suppresses the icon.
* **Verified:** Header **11 passed** (incl. the new test) + MultiAgentLogo **3 passed**, eslint clean. The change is an edit to an existing file, so Vite **HMR applies it without a restart** — refresh the browser.
* **Lesson (also in repo memory):** never pass Fluent `Avatar` a `name` when you want a custom logo to show — `name` initials always win over `icon`.

## BUG‑0073 — delete document must also remove the source blob (implementation)

Planned (analysis-first) then implemented the fix so an admin document delete removes the source blob alongside its index/pgvector chunks. Confirmed from code that [delete_document_endpoint](../src/backend/routers/admin.py) only called `search.delete_by_source` (pgvector `DELETE … WHERE title=$1` / Azure Search `delete_documents`) — the Storage **blob** was never deleted, so [GET /api/files/{filename}](../src/backend/routers/files.py) kept streaming the "deleted" file. In this deployment the index **is** pgvector (one store), so `delete_by_source` already clears it; the gap was the blob.

* **Unit A** — added `delete_document(filename, *, settings, credential) -> bool` to [services/files.py](../src/backend/services/files.py), mirroring `download_document`: `ContainerClient.delete_blob`, `ResourceNotFoundError` → `False` (idempotent), other `AzureError` → log + re-raise (Hard Rule #14), `_validate_filename` rejects URL/path-typed sources (`ValueError`) before any SDK call. 4 tests in [test_services_files.py](../tests/backend/test_services_files.py).
* **Unit B** — wired it into the endpoint: added `SettingsDep` + `CredentialDep`; after `delete_by_source`, when `documents_container` is set, call `delete_document` (swallowing `ValueError` for URL sources → `blob_deleted=False`); now 404s only when **neither** an index chunk **nor** a blob existed. Extended `DeleteDocumentResponse` with `blob_deleted: bool` (default `False`, non-breaking for the FE OpenAPI client). Updated the two existing delete tests + added blob-only-removed, URL-source-skip, and no-container-skip cases; the production-auth test needed a `get_credential` stub (the new `CredentialDep` resolves before the auth gate, but a real deploy has the lifespan credential so the 401 still fires).
* **Verified:** 134 admin + files-service tests passed, pyright on the three changed source files **0/0/0**, AST gates **355 passed**.
* **Closed (operator decision):** marked [BUG‑0073](../bugs.md) **fixed** on the strength of the implementation + test coverage. Live end-to-end verification (delete a real document, confirm `GET /api/files/<src>` 404s) was deferred — it can be confirmed during the next local run. Next in the sweep: BUG‑0074.

* **Unit 2** — threaded `userInfo` through [Header.tsx](../src/frontend/src/components/Header/Header.tsx) → [App.tsx](../src/frontend/src/App.tsx) (`userInfo={auth.userInfo}` from `useAuth`).
