## Summary

Landed and live-verified the fix for **BUG-0028** — the `langgraph` orchestrator returning the out-of-domain fallback ("The requested information is not available in the retrieved data…") for in-domain queries. With operator consent (Hard Rule #10 — the change edits the BUG-0011-protected shared CWYD safety prompt), softened the strict ported-v1 grounding language in `CWYD_GUARDRAIL` + `CWYD_DEFAULT_BODY` ([definitions.py](../../src/backend/core/agents/definitions.py)) to a relevance-based "answer when the documents are relevant, even if brief" directive, while keeping an explicit ungrounded-creative-content ban, the exact fixed fallback string, the `[doc+index]` citation format, and the `compose_cwyd_instructions` guardrail-wrapping invariant. Landed test-first (four new invariant assertions). Live-verified on `langgraph` + `gpt-5.1`: the previously on-the-boundary "tell me about employee benefits" now grounds 3/3, "what health plans are available?" grounds, and "what is the capital of France?" + "write me a poem about dragons" both return the fixed fallback.

The BUG-0028 *secondary* improvement — threading the configured `openai_temperature` / `openai_max_tokens` through `BaseLLMProvider.complete()` — was attempted as a separate unit and **reverted**: passing `max_tokens` made every `langgraph` chat `500` because `gpt-5.1` rejects `max_tokens` (it requires `max_completion_tokens`). Filed that as **BUG-0029** and deferred the threading behind it.

## Bugs

* **BUG-0028 — fixed (2026-06-14).** `langgraph` over-refused in-domain queries because the strict ported-v1 `CWYD_GUARDRAIL` "not enough information → fallback" clause made `gpt-5.1` over-conservative on the raw chat-completion path (retrieval was healthy; `agent_framework` survives the same prompt because Foundry IQ presents KB results as authoritative tool outputs). In `postgresql` mode `langgraph` is the only path, so this blocked answering entirely. Fixed by the prompt softening above; live-verified. Registry row + detail Resolution paragraph updated in [bugs.md](../bugs.md).
* **BUG-0029 — filed (open).** `FoundryIQ.chat()` / `chat_stream()` send the Chat Completions `max_tokens` parameter, which `gpt-5.1` (and gpt-5 / o-series chat models) reject with `400` (`Unsupported parameter: 'max_tokens' … Use 'max_completion_tokens' instead`). Latent in the shipped tree because `complete()` doesn't thread `max_tokens`; surfaced only when the BUG-0028 secondary threading was added (and reverted). Fix (separate unit): send `max_completion_tokens` in the provider, then re-attempt the `complete()` threading — but first confirm `gpt-5.1` accepts a custom `temperature` (gpt-5 reasoning models constrain it).

## Decisions

* **Softened the shared CWYD prompt rather than branching per-orchestrator** (operator choice). Both orchestrators share `compose_cwyd_instructions(CWYD_DEFAULT_BODY)`; the softer grounding therefore applies to `agent_framework` too, which is acceptable because it keeps the identical creative-content ban + fixed fallback. The single broad "employee benefits" overview previously sat on gpt-5.1's grounding decision boundary; with the softened wording it grounded 3/3 in live testing (caveat accepted: a very small corpus can still produce boundary variance).
* **Reverted, not pursued, the temperature/max_tokens threading.** The user's selected approach allowed deferring it ("unless deferred"). It is not required for the BUG-0028 fix, and doing it correctly needs the BUG-0029 `max_completion_tokens` provider fix first. Kept the change minimal — only the two prompt constants ship.
