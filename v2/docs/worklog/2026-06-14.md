## Implementation — BUG-0013 substantive reasoning (modular design; unit 1 landed 2026-06-14)

Picked up BUG-0013 in numeric order. Read the full reasoning path end-to-end and recorded the confirmed root cause in [bugs.md](../bugs.md): the reasoning **plumbing is complete and correct** on both orchestrators (FoundryIQ `reason()` already streams Responses-API summary deltas; `langgraph` forwards `reasoning` events verbatim; `agent_framework._update_to_events` already maps `text_reasoning` blocks → `reasoning`; the FE panel renders them) — but **no substantive reasoning content is produced** because the *answer* model is never asked for a summary on either live path (`langgraph` answers via `complete()` → `chat()` Chat Completions; `agent_framework` runs `CWYD_AGENT` with no reasoning knob).

**Operator constraint:** the fix must cover **both** orchestrators and stay modular for any future orchestrator. Design adopted (no per-orchestrator reasoning logic duplicated):

* **Shared seam already exists** — every orchestrator forwards `reasoning`-channel `OrchestratorEvent`s. The only missing piece is a single switch that tells each model-invocation boundary to *request* a reasoning summary.
* **One shared knob → consumed at each model boundary.** `OpenAISettings.reasoning_summary` (env `AZURE_OPENAI_REASONING_SUMMARY`, default off) is the single source of truth. `FoundryIQ.complete()` reads it → streams the answer via the Responses API with `reasoning.summary="auto"` (covers `langgraph` + any future orchestrator that answers through `self.llm.complete()` with zero extra code). `agent_framework.run()` reads it → requests reasoning on the agent run (lights up its existing `text_reasoning` forwarding).
* **Modularity guarantee:** a new orchestrator gets reasoning either free (answers via `self.llm.complete()`) or by reading the same one knob + forwarding reasoning events (already standardized by `OrchestratorBase`).
* **Default off** because the Responses API rejects the `reasoning` parameter for non-reasoning deployments (e.g. gpt-4o); the operator's gpt-5.1 deployment opts in via env.

### Done

* **Unit 1 — shared knob (test-first).** Added `reasoning_summary: bool = False` to `OpenAISettings` in [settings.py](../../src/backend/core/settings.py) with a docstring stating it is read by every model-invocation boundary and why it defaults off. Added two tests in [test_settings.py](../../tests/backend/core/test_settings.py) — defaults-off and `AZURE_OPENAI_REASONING_SUMMARY=true` override-enables. `test_settings.py` **66/66** green; `pyright --strict` **0/0/0** on `settings.py`.

### Done (continued)

## Investigation — reasoning shows two blocks on agent_framework vs one on langgraph (2026-06-14)

Operator: *"in agent framework the thinking/reasoning appears twice … why do agent_framework and langgraph behave different, it should be the same."* The example showed two reasoning sections — **Searching for employee benefits** (pre-search) then **Summarizing health benefits** (post-search).

Read both reasoning paths end-to-end. **Root cause is architectural, not a duplication bug:**

* **`langgraph`** ([langgraph.py](../../src/backend/core/providers/orchestrators/langgraph.py)) is **pre-retrieval RAG**: it retrieves KB docs first, injects them as a `[docN]` system block, then the model reasons **once** over the already-present context → one reasoning summary.
* **`agent_framework`** ([agent_framework.py](../../src/backend/core/providers/orchestrators/agent_framework.py)) is an **agentic tool-calling loop**: the model reasons (turn 1: *"I'll call knowledge_base_retrieve"*), the server-side Foundry IQ KB MCP tool runs, then the model reasons **again** (turn 2: *"summarize the document"*) → two reasoning summaries, one per agentic turn around the tool call.

Confirmed it is **not** within-turn `delta`+`done` double-emit: the OSS `agent_framework_openai` chat client dedupes per turn via a `seen_reasoning_delta_item_ids` set ([_chat_client.py](../../.venv/Lib/site-packages/agent_framework_openai/_chat_client.py) line 615), so the `reasoning_summary_text.done` full-text frame is suppressed when deltas were seen. The two blocks are two genuine agentic turns. Both orchestrators render reasoning in the same single `<details>` panel; the visible difference is the **number of reasoning summaries the model produces**, which is downstream of the retrieval architecture.

**Operator direction (2026-06-14):** *"drop any extra wording added (searching and summarizing) and add break between them blocks"* and *"be sure langgraph and agent_framework use the same logic, I don't want duplicated logic for this."* Decided **not** to touch either orchestrator's event emission — the two feeds converge only at full-text granularity in the FE renderer (langgraph streams char-level summary deltas where a bold header is split across fragments and can't be stripped per-event; agent_framework emits per-turn blocks). A single shared FE formatter is therefore the *only* one-logic-path option, and it is stronger than a backend seam that would risk divergence from the granularity mismatch.

**Resolution — `formatReasoning` shared FE formatter (done).** Added [reasoningText.tsx](../../src/frontend/src/pages/chat/components/reasoningText.tsx): a pure `formatReasoning(parts: string[]): string` that joins the reasoning array, drops the model's standalone bold section titles (`**Title**` on its own line — regex `/\s*\*\*[^*\n]+\*\*[ \t]*\n+/g`), collapses the resulting 3+ newlines to a single blank-line break, and trims. The langgraph char-level delta stream carries no standalone titles, so it round-trips as the verbatim join (existing behavior preserved); the agent_framework headered blocks render with titles stripped and bodies break-separated. Inline bold not followed by a newline survives. [MessageList.tsx](../../src/frontend/src/pages/chat/components/MessageList.tsx) now renders `formatReasoning(m.reasoning)` instead of `m.reasoning.join("")` — both orchestrators flow through the one helper, so there is no duplicated per-orchestrator logic. Test-first: 7 unit cases in [reasoningText.test.tsx](../../src/tests/frontend/pages/chat/components/reasoningText.test.tsx) (verbatim join, header strip + break, jammed second title, adjacent-title collapse, inline-bold preserved, empty) + 1 integration case in [MessageList.test.tsx](../../src/tests/frontend/pages/chat/components/MessageList.test.tsx). All 36 tests green; `tsc --noEmit` clean on both FE projects. The running `npm run dev` server hot-reloads it.
