## Implementation — BUG-0013 substantive reasoning (modular design; unit 1 landed 2026-06-14)

Picked up BUG-0013 in numeric order. Read the full reasoning path end-to-end and recorded the confirmed root cause in [bugs.md](../bugs.md): the reasoning **plumbing is complete and correct** on both orchestrators (FoundryIQ `reason()` already streams Responses-API summary deltas; `langgraph` forwards `reasoning` events verbatim; `agent_framework._update_to_events` already maps `text_reasoning` blocks → `reasoning`; the FE panel renders them) — but **no substantive reasoning content is produced** because the *answer* model is never asked for a summary on either live path (`langgraph` answers via `complete()` → `chat()` Chat Completions; `agent_framework` runs `CWYD_AGENT` with no reasoning knob).

**Operator constraint:** the fix must cover **both** orchestrators and stay modular for any future orchestrator. Design adopted (no per-orchestrator reasoning logic duplicated):

* **Shared seam already exists** — every orchestrator forwards `reasoning`-channel `OrchestratorEvent`s. The only missing piece is a single switch that tells each model-invocation boundary to *request* a reasoning summary.
* **One shared knob → consumed at each model boundary.** `OpenAISettings.reasoning_summary` (env `AZURE_OPENAI_REASONING_SUMMARY`, default off) is the single source of truth. `FoundryIQ.complete()` reads it → streams the answer via the Responses API with `reasoning.summary="auto"` (covers `langgraph` + any future orchestrator that answers through `self.llm.complete()` with zero extra code). `agent_framework.run()` reads it → requests reasoning on the agent run (lights up its existing `text_reasoning` forwarding).
* **Modularity guarantee:** a new orchestrator gets reasoning either free (answers via `self.llm.complete()`) or by reading the same one knob + forwarding reasoning events (already standardized by `OrchestratorBase`).
* **Default off** because the Responses API rejects the `reasoning` parameter for non-reasoning deployments; the operator's gpt-5.1 deployment opts in via env.

* **Unit 1 — shared knob (test-first).** Added `reasoning_summary: bool = False` to `OpenAISettings` in [settings.py](../../src/backend/core/settings.py) with a docstring stating it is read by every model-invocation boundary and why it defaults off. Added two tests in [test_settings.py](../../tests/backend/core/test_settings.py) — defaults-off and `AZURE_OPENAI_REASONING_SUMMARY=true` override-enables. `test_settings.py` **66/66** green; `pyright --strict` **0/0/0** on `settings.py`.

## Implementation — BUG-0030 A0 live capture (Phase 0 hard gate, 2026-06-14)

Ran the Phase 0 live capture against the real `agent_framework` deployment (throwaway integration-lane probe, created and deleted; it forced the orchestrator past any persisted cosmos override, monkeypatched `citations_from_annotations` to record its raw input, and wrote the answer text + annotations to a UTF-8 file so the Windows cp1252 console never had to encode `【`). ### Decisive result — the divergence DOES reproduce (supersedes the earlier "does not reproduce" note)

The current live `agent_framework` path returns the **un-normalized** shape on both fronts:

* **Answer text carries native markers.** The buffered answer literally contains `【6:1†source】` and `【6:0†source】` (`\u3010`=`【`, `\u2020`=`†`, `\u3011`=`】`), e.g. `...through Northwind Health【6:1†source】:` and `...Employee +2 or more: $78.00【6:0†source】`. This is exactly the operator's report and the opposite of the earlier integration-lane note in [bugs.md](../bugs.md) that recorded a clean `[doc1]` answer. The earlier capture and this one disagree because the friendly-shape output is **Foundry-IQ-KB-config-dependent** and the live KB now emits the native shape.
* **Citation objects are raw.** `citations_from_annotations` currently returns `id == title == url == mcp://searchindex/<doc-key>`, `snippet == ""`, `score == null`. No filename, no chunk text.

### The mapping is offset-anchored, NOT `N:M`-based (key finding)

Each SDK annotation carries `annotated_regions = [{type: "text_span", start_index, end_index}]` — the **exact character span** of one native marker in the answer text. Verified: the first region `start_index: 78, end_index: 90` lands precisely on the first `【6:1†source】` (each marker is exactly 12 chars), and the offsets align with the buffered `content` string 1:1. `citations_from_annotations` already groups annotations by source id and aggregates every span under `Citation.metadata["annotated_regions"]` (Citation[0] = 15 spans, Citation[1] = 9 spans).

The `N:M` inside `【N:M†source】` is **misleading** and must not be parsed: `N` is constant (`6`) and `M` is the KB's internal source index, **not** the final citation order. In this capture `M=1` belongs to Citation[0] and `M=0` belongs to Citation[1] — inverted from any naive `M`→`docN` reading. The correct mapping uses the **citation grouping order** (first-occurrence order the SDK already produces): Citation[0] → `[doc1]`, Citation[1] → `[doc2]`, and each citation's `annotated_regions` spans are the marker occurrences to rewrite to that citation's `[docN]`.

### Plan-altering finding — friendly title + snippet are NOT in the annotation

The full `raw_representation` is `{type: "url_citation", start_index, end_index, title: "mcp://searchindex/<doc-key>", url: "mcp://searchindex/<doc-key>"}` and `additional_properties` is only `{annotation_index: <n>}`. There is **no filename and no chunk text anywhere** in the annotation. So the planned A3 recovery ("snippet → `raw_representation` → `additional_properties`") cannot work on the current KB config — the data simply isn't there. Friendly title + snippet require either (a) a Foundry IQ KB citation-field-mapping change so the KB emits them (infra, the long-term fix the earlier capture proves is possible), or (b) a secondary lookup resolving `mcp://searchindex/<doc-key>` → title + content via Azure Search in our code, or (c) accepting `[docN]` id with no friendly title/snippet (degraded — loses the right-panel snippet). This fork is surfaced to the operator before any A3 / FE-snippet work.

### Consequences

* The existing live guard `test_in_domain_citation_ids_use_normalized_docn_shape` now fails against this deployment (its `title.startswith("mcp://")` / non-`[docN]` id assertion is exactly this regression) — confirming the guard works and the regression is real.
* The answer-text + citation-id normalizer (next unit) is now fully unblocked and deterministic: rewrite each `Citation.metadata["annotated_regions"]` span to the citation's 1-based `[docN]`, renumber citation ids to `[docN]`, and strip any residual native marker. It does **not** depend on the title/snippet fork, so it lands independently.

## Implementation — BUG-0030 A2 offset-anchored normalizer (landed 2026-06-14)

First code unit after the A0 gate: a pure `normalize_kb_citations(answer, citations) -> tuple[str, list[Citation]]` in [citations.py](../../src/backend/core/tools/citations.py).

### Done

* **Unit — `normalize_kb_citations` (test-first).** Renumbers each citation to its 1-based `[docN]` and rewrites every native `【N:M†source】` marker the SDK attributed to it (via `Citation.metadata["annotated_regions"]`) to that `[docN]`. Mapping is **offset-anchored** — for each region it slices `answer[start:end]` and rewrites only when the slice `fullmatch`es the native-marker regex `【[\d:]+†[^】]+】`, so the misleading `N:M` is never parsed and an offset-drift region can never corrupt non-marker text. After the attributed markers are mapped, a defensive `re.sub` strips any residual native marker so it never reaches the UI. A private `_region_bounds` helper validates each span is a well-formed, non-inverted integer mapping. Title / url / snippet are intentionally left untouched (the raw `mcp://` id stays on `title`) — friendly-title recovery is the separate operator-gated fork.
* **Tests.** Seven cases in [test_citations.py](../../tests/backend/core/tools/test_citations.py): grouping-order mapping (proves `M=1`→`[doc1]`, `M=0`→`[doc2]`, i.e. **not** `N:M`), every-repeat rewrite, residual-orphan strip, id renumber when the answer has no native markers, empty-citations strip, offset-drift no-corruption, and title/url/snippet/score preservation. Offsets are computed with `answer.index(marker)` so the fixtures can't drift.
* **Validation:** `test_citations.py` 29/29 green; `pyright --strict` clean on `citations.py` (the SDK-boundary `Mapping`/`list` reads are narrowed with one commented `cast` per Hard Rule #11); shared AST gates green (955 passed, 1 skipped — the process-narrative + imports-at-top gates pass with the new Unicode-marker comment and `cast`/`Mapping` imports).

### Result
