## Summary

Landed and live-verified the fix for **BUG-0028** — the `langgraph` orchestrator returning the out-of-domain fallback ("The requested information is not available in the retrieved data…") for in-domain queries. With operator consent (Hard Rule #10 — the change edits the BUG-0011-protected shared CWYD safety prompt), softened the strict ported-v1 grounding language in `CWYD_GUARDRAIL` + `CWYD_DEFAULT_BODY` ([definitions.py](../../src/backend/core/agents/definitions.py)) to a relevance-based "answer when the documents are relevant, even if brief" directive, while keeping an explicit ungrounded-creative-content ban, the exact fixed fallback string, the `[doc+index]` citation format, and the `compose_cwyd_instructions` guardrail-wrapping invariant. Live-verified on `langgraph` + `gpt-5.1`: the previously on-the-boundary "tell me about employee benefits" now grounds 3/3, "what health plans are available?" grounds, and "what is the capital of France?" + "write me a poem about dragons" both return the fixed fallback.

## Bugs

* **BUG-0028 — fixed (2026-06-14).** `langgraph` over-refused in-domain queries because the strict ported-v1 `CWYD_GUARDRAIL` "not enough information → fallback" clause made `gpt-5.1` over-conservative on the raw chat-completion path (retrieval was healthy; `agent_framework` survives the same prompt because Foundry IQ presents KB results as authoritative tool outputs). * **BUG-0029 — filed (open).** `FoundryIQ.chat()` / `chat_stream()` send the Chat Completions `max_tokens` parameter, which `gpt-5.1` (and gpt-5 / o-series chat models) reject with `400` (`Unsupported parameter: 'max_tokens' … Use 'max_completion_tokens' instead`). Latent in the shipped tree because `complete()` doesn't thread `max_tokens`; surfaced only when the BUG-0028 secondary threading was added (and reverted). Fix (separate unit): send `max_completion_tokens` in the provider, then re-attempt the `complete()` threading — but first confirm `gpt-5.1` accepts a custom `temperature` (gpt-5 reasoning models constrain it).

## Investigation — orchestrator parity review (citations + reasoning)

Operator compared the two orchestrators answering the same "tell me about employee benefits" question and flagged three parity gaps. Reviewed the code paths end-to-end; findings:

* **Citation shape diverges per orchestrator — NEW defect, filed BUG-0030 (high).** `langgraph` produces inline `[doc1]` + a friendly `Benefit_Options.pdf` source label; `agent_framework` produces inline `【6:0†source】` (OpenAI Responses native annotation format) + a raw `mcp://searchindex/<doc-key>` source label. Root cause: `langgraph` injects a `[docN]: <content>` sources block into the prompt and post-processes the `[docN]` markers (`filter_to_referenced` → `Citation(id="[doc1]", title=filename)`), so the shared prompt's `[docN]` rule is honored; `agent_framework` grounds via the server-side Foundry IQ KB MCP tool, which makes the model emit its own native annotations and surfaces the raw document key (not the filename) — `citations_from_annotations` passes that key straight through. The system **instructions** are identical for both (`compose_cwyd_instructions(...)`), but the **citation mechanism** is in-prompt vs server-side-tool, so the effective behavior diverges. Distinct from the FE-only BUG-0014/0015/0016 (all of which assume `[docN]` already exists). Fix direction recorded in BUG-0030: normalize the `agent_framework` path in post-processing — rewrite `【N:M†source】` → `[docN]` and resolve the friendly filename — so both orchestrators emit one `[docN]` + filename shape. Needs operator direction (open question: what the Foundry IQ annotation carries for `title`).
* **Instruction composition is not a shared seam — NEW defect, filed BUG-0031 (medium).** Follow-up to the operator's "shared code, not just identical strings" challenge: confirmed the two orchestrators do **not** share instruction-building *code* — they are two parallel derivations that converge only on the default prompt. With **no override**, both resolve to `CWYD_AGENT.instructions` = `compose_cwyd_instructions(CWYD_DEFAULT_BODY)` — identical, guardrail-wrapped (so current default-prompt testing is unaffected). With an **operator override** they diverge: `agent_framework` wraps the authored text via `_resolve_definition` → `compose_cwyd_instructions(text)` (the `CWYD_GUARDRAIL` bookends it), but `langgraph` injects `effective.cwyd_agent_instructions` — the **raw, un-wrapped** stored override (the PATCH handler stores it raw; `resolve_effective_config` overlays that raw value) — directly as a SYSTEM message with no wrap. Net: an admin custom prompt keeps the guardrail (out-of-domain refusal + `[docN]` + fixed fallback) on `agent_framework` but **drops it on langgraph**, so BUG-0011's "persona between the guardrail bookends" invariant holds on only one path. RAI still screens authored prompts at write time, so BUG-0011's malicious-jailbreak vector is **not** reintroduced — only the standing guardrail behaviors are lost on langgraph. Same shared-seam fix family as BUG-0030: compose the wrapped instructions once (in `resolve_effective_config` or the router) so both orchestrators consume identical text.
* **Duplicated "Searching the knowledge base…" narration — already tracked: BUG-0027 (in-progress), un-finished half.** On the `agent_framework` path the narration fires **twice**: once upfront from the shared `run_chat` pipeline (`retrieval_hint=KB_SEARCH_NARRATION`, gated on a wired search backend — the landed BUG-0027 part-1), and again from `agent_framework._update_to_events` when the `mcp_server_tool_call` block surfaces (the BUG-0026 fix). BUG-0027's own row already prescribes the remaining work — *"then drop the now-redundant late per-call narration from `agent_framework._update_to_events` (keep the `tool` event)"* — which has not landed, so the duplication is the open half of BUG-0027, **not** a new bug. No new bug filed.
* **No real LLM reasoning in the thinking panel — already tracked: BUG-0013 (open).** The reasoning panel only ever shows the canned `KB_SEARCH_NARRATION`, never model reasoning, because the chat path runs `gpt-5.1` (a non-reasoning chat model) on both orchestrators — there is no `text_reasoning` content to emit. Surfacing genuine model reasoning needs the o-series `reasoning_deployment` (o4-mini) routing, which is the BUG-0013 concern. Expected behavior with a non-reasoning model; no new bug filed.
