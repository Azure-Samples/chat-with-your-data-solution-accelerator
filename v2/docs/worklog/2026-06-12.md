## Decisions

* `BUG-0021` is fixed with the **MACAE GA agent pattern**, settled with the operator: get-or-create a name-addressed Foundry **Prompt Agent** via `AIProjectClient.agents.create_version` + `PromptAgentDefinition`, store the agent **name** in the DB (matching MACAE), and invoke that same named agent at chat time. This removes the Assistants-API (`asst_`) vs Responses-API-named-agent split that produced the `404`.
* **KB grounding moves server-side.** The agent definition carries an MCP tool (`server_url=.../knowledgebases/<kb>/mcp`, `allowed_tools=["knowledge_base_retrieve"]`, `require_approval="never"`, `project_connection_id=<connection_name>`), so Foundry IQ runs retrieval under the Project identity. This retires the per-request bearer + caller-owned `httpx` client introduced for `BUG-0020` (the `401` connect fix), which becomes unnecessary once the connect is server-side; the dead helper is removed in U10 only if fully unused.
* **Name-based identity.** `get_or_create_agent` now returns and persists the agent *name*; the `409`-on-create path re-reads and reuses the existing named agent (idempotent), never surfacing the conflict as an error.
* **Citation construction is shared.** A new seam maps `agent_framework` content annotations to the common `Citation` model so both the `langgraph` and `agent_framework` orchestrators emit identical `citation` SSE events (`BUG-0013`/`0015`/`0016` parity).
* The RAI agent uses the **same** GA pattern and a strong (operator-configured) deployment; its `deployment_attr` is unchanged.
