## Summary

Started the structural fix for `BUG-0021` (the `agent_framework` chat path `404`s with `Agent 'cwyd' not found`). The operator chose the MACAE GA agent pattern: provision a name-addressed Foundry **Prompt Agent** through the GA control plane (`AIProjectClient.agents.create_version` with a `PromptAgentDefinition`), persist the agent **name** (not the `asst_` id), and invoke that same named agent at chat time — closing the two-disjoint-resource-types gap that caused the `404`. The Knowledge Base grounding moves server-side: the agent definition carries an MCP tool that Foundry IQ executes under the Project identity, retiring the per-request bearer + caller-owned `httpx` client added for `BUG-0020`. The RAI agent migrates to the same pattern, and citation construction is factored into a seam shared by the `langgraph` and `agent_framework` orchestrators. Landed U1–U4 plus the first half of U5: the Foundry Project↔Search connection-name setting (U1), its Bicep wiring (U2), the core lifecycle rewrite of `BaseAgentsProvider.get_or_create_agent` to the GA get-or-create flow with name-based identity and a 409-race guard (U3), the `build_agent` DRY seam composing a client-side `agent_framework.Agent` over `FoundryChatClient` (U4), and the `agent_framework` orchestrator rewrite (U5 part 1) — named agent via `build_agent`, server-side KB grounding via an `MCPTool`, async-CM agent lifecycle, and retirement of the per-request bearer + `httpx` client — with both affected test suites migrated, then the U5 router wiring (part 2): `conversation.py` now forwards `agents=` / `db=` into the single registry dispatch and no longer bootstraps the agent (create-if-missing moved into the provider's `build_agent`). **Flagged:** the backend `pyright --strict` gate is RED on a pre-existing `base.py:258` type error (`PROMPTDEF-TOOLS-TYPE-DEBT`), recorded as debt for an operator design call.

## Decisions

* `BUG-0021` is fixed with the **MACAE GA agent pattern**, settled with the operator: get-or-create a name-addressed Foundry **Prompt Agent** via `AIProjectClient.agents.create_version` + `PromptAgentDefinition`, store the agent **name** in the DB (matching MACAE), and invoke that same named agent at chat time. This removes the Assistants-API (`asst_`) vs Responses-API-named-agent split that produced the `404`.
* **KB grounding moves server-side.** The agent definition carries an MCP tool (`server_url=.../knowledgebases/<kb>/mcp`, `allowed_tools=["knowledge_base_retrieve"]`, `require_approval="never"`, `project_connection_id=<connection_name>`), so Foundry IQ runs retrieval under the Project identity. This retires the per-request bearer + caller-owned `httpx` client introduced for `BUG-0020` (the `401` connect fix), which becomes unnecessary once the connect is server-side; the dead helper is removed in U10 only if fully unused.
* **Name-based identity.** `get_or_create_agent` now returns and persists the agent *name*; the `409`-on-create path re-reads and reuses the existing named agent (idempotent), never surfacing the conflict as an error.
* **Citation construction is shared.** A new seam maps `agent_framework` content annotations to the common `Citation` model so both the `langgraph` and `agent_framework` orchestrators emit identical `citation` SSE events (`BUG-0013`/`0015`/`0016` parity).
* The RAI agent uses the **same** GA pattern and a strong (operator-configured) deployment; its `deployment_attr` is unchanged.
