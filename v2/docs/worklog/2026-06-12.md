## BUG-0013 — reasoning feed empty (design locked + Unit 1)

### Design (locked with operator)

The operator chose **model-agnostic activity narration _and_ explicit reasoning-model detection/branching** — both, not either. Governing constraints (operator): no hard-coded model names (the deployment must stay swappable), content-reactive, and account for **Foundry IQ owning retrieval server-side** (narrate from observable KB activity).

Root cause (confirmed against the post-`BUG-0021` code): the `text_reasoning → reasoning` SSE pipeline is already fully wired — `_update_to_events` maps `text_reasoning` content blocks onto `OrchestratorChannel.REASONING`, and the FE renders them in the collapsible "Thinking" panel (panel shows when `m.streaming || m.reasoning.length > 0`). The gap is upstream: `agent.run(...)` requests **no** reasoning output, so the chat model emits no `text_reasoning` summary blocks → the panel stays empty. Foundry IQ runs KB retrieval (query planning, ranking) **server-side**, so the only client-observable signals of that work are the KB `function_call` block and the returned citation annotations.

### Plan (one unit per turn, test-first, in dependency order)

* **U1 — KB-search activity narration (model-agnostic).** Emit a `reasoning`-channel narration when the agent invokes KB retrieval (the `knowledge_base_retrieve` `function_call`), immediately before the raw `tool` event for that call. Guarantees a non-empty "Thinking" panel for *any* model, independent of native reasoning summaries. **This unit.**
* **U2 — sources-found narration.** After the stream, emit a `reasoning`-channel narration summarizing the resolved citation count, in the `run()` citation section.
* **U3 — explicit reasoning-capability declaration.** Add a declared, **per-agent** reasoning-effort field to the agent-definition layer (`AgentDefinition`) — a `ReasoningEffort` `StrEnum` (`NONE`/`LOW`/`MEDIUM`/`HIGH`), defaulting `NONE`. Explicit (scenario/operator declares it), model-agnostic (no name matching), forward-compatible (swap the deployment + set the field). Per-agent (not a global `OpenAISettings` knob) so the RAI classifier is never forced to reason. CWYD declares a non-`NONE` effort; RAI stays `NONE`.
* **U4 — orchestrator branching.** When the active agent declares a non-`NONE` reasoning effort, thread the reasoning request into `agent.run(...)` via the SDK's supported reasoning option so a reasoning-capable model emits `text_reasoning` summaries; otherwise pass nothing (safe for non-reasoning models). ## Conversation regression — root cause + MACAE grounding review (decision pending)

The "tell me about employee benefits" turn returns the out-of-domain fallback ("The requested information is not available in the retrieved data…") instead of answering. Root-caused to **two independent causes, one per orchestrator**:

* **`agent_framework` (Foundry IQ path):** the KB MCP tool is never built — `_build_kb_tool()` returns `None` because `AZURE_AI_SEARCH_CONNECTION_NAME` is empty in the env **and** the Knowledge Base `cwyd-kb` does not exist in the search service (`404` on `/knowledgebases('cwyd-kb')`). * **`langgraph` (legacy direct-search path):** search is healthy (5 hits, ~6.2 KB sources block injected), but the ported v1 strict CWYD prompt makes `gpt-5.1` over-refuse — proven by prompt-variant probes (the guardrail alone triggers the refusal; a minimal prompt + the same sources answers correctly).

The operator reframed the fix as a **Foundry IQ wiring/migration** concern (ADR 0025 moved grounding server-side), not a prompt/model issue, and asked to review how **MACAE** structures grounding to pick the more modular option.

### MACAE grounding architecture (`data/sample_code/macae/src/backend`)

Grounding in MACAE is **config-driven on the agent, not hard-coded in the orchestrator**. `agents/agent_template.py::AgentTemplate.__init__` accepts three optional, composable grounding configs (`config/mcp_config.py`), and `_build_tools()` assembles whichever are present:

| Config | Tool built | Source of truth | Provisioning |
|---|---|---|---|
| `MCPConfig` | `MCPTool` / client `MCPStreamableHTTPTool` | any MCP server | an MCP server (custom tools, not RAG) |
| `VectorStoreConfig` | `FileSearchTool(vector_store_ids=[id])` | **Foundry-managed vector store** (OpenAI `vector_stores` API) | a vector store (resolve name→id) |
| `KnowledgeBaseConfig` | `MCPTool(server_url=…/knowledgebases/<kb>/mcp, project_connection_id=<conn>)` | **Azure AI Search** via Foundry IQ KB | KB + knowledge source + CognitiveSearch connection |

MACAE also **bakes the KB `MCPTool` into the `PromptAgentDefinition`** (portal-visible, persisted) and reconciles a stale KB name / connection on each `open()`; CWYD attaches its KB tool **per-request** as an `extra_tool` instead. CWYD's current KB path already produces byte-for-byte the same `MCPTool` shape MACAE's `kb_config` does (same URL pattern, `allowed_tools=["knowledge_base_retrieve"]`, `require_approval="never"`, `project_connection_id`). The two real differences: (a) MACAE offers **FileSearch + managed vector stores** as a second RAG strategy; (b) MACAE makes the strategy a **config choice**, CWYD hard-codes one strategy in `agent_framework.py`.

### The "more modular option" — two levels

* **Mechanism (which RAG backend):**
  * **A. KB MCP (CWYD today):** keeps Azure AI Search as the index of record; the Phase 6 ingestion pipeline already targets it. Fix = wire the three gaps (seed `cwyd-kb` + `cwyd-index-ks`, set `AZURE_AI_SEARCH_CONNECTION_NAME` to the `search-srch-<SUFFIX>` connection, run on the `agent_framework` orchestrator). Lowest risk; no ingestion rework.
  * **B. FileSearch + managed vector stores (MACAE `VectorStoreConfig`):** simplest agent wiring (just a vector-store name, no KB/knowledge-source/connection dance), Foundry owns the store. **But** it abandons the Azure AI Search index and forces re-ingesting documents into Foundry vector stores — a re-architecture of the Phase 6 ingestion pipeline. More modular at the agent level, disruptive below it.
* **Architecture (CWYD-native expression of MACAE's lesson):** make **grounding a registry-backed provider domain** (Hard Rule #4) — e.g. `providers/grounding/` with `kb_mcp` and `file_search` registered, selected by config; the orchestrator asks the grounding provider for its tool(s) instead of calling `_build_kb_tool()` itself. This makes A↔B a swappable config exactly like MACAE's `_build_tools()`, keeping the orchestrator both model-agnostic and grounding-agnostic. It is a **structural change (new provider domain) → Hard Rule #10 (operator confirmation required)**.

### Recommendation (pending operator decision)

* Keep **Option A (KB MCP)** as the grounding mechanism — it preserves Azure AI Search + the Phase 6 ingestion pipeline, and CWYD is one connection-name + one KB-seed away from working.
* Adopt the **registry-backed grounding provider** as the modular shape so FileSearch can be added later as a second registered strategy without touching the orchestrator — only on explicit operator go-ahead (Hard Rule #10).
* Decide `langgraph`'s fate separately: fix its over-strict prompt as the no-Foundry-IQ fallback, or retire it.

**Status:** awaiting operator decision on (1) grounding mechanism A vs B, (2) whether to introduce the grounding provider domain now, (3) `langgraph` prompt fix vs retire.

### Operator decision + the PostgreSQL question (resolved by existing architecture)

**Operator chose the MACAE / Foundry IQ option** — retrieval must run **server-side in Foundry IQ**, not in our code. Then asked: *what happens to PostgreSQL?*

Answer, grounded in the shipped Phase 8 architecture (ADR 0021 + ADR 0022; dev_plan §0.1 "pgvector constraint"): **Foundry IQ has no pgvector knowledge source.** Foundry IQ Knowledge Bases sit **only** on Azure AI Search. So the platform — not our code — forces a two-path split, and that split is already built and guarded:

| Mode (hard-coupled by `_enforce_mode_consistency`) | Orchestrator | Retrieval | Search logic location |
|---|---|---|---|
| `cosmosdb` ↔ `AzureSearch` | `agent_framework` | Foundry IQ KB MCP (`knowledge_base_retrieve`) | **server-side in Foundry IQ** ← the MACAE option |
| `postgresql` ↔ `pgvector` | `langgraph` | `PgVector.search()` (cosine / FTS) | **in our code** (unavoidable) |

Consequences of the operator's decision:

1. **CosmosDB (the live env):** the MACAE option is *already implemented* as the `agent_framework` orchestrator. The conversation is broken only because of three wiring gaps — `AZURE_AI_SEARCH_CONNECTION_NAME` unset, KB `cwyd-kb` not seeded into `srch-…`, orchestrator pinned to `langgraph`. 2. **PostgreSQL:** **cannot** move to Foundry IQ — there is no pgvector knowledge source (Azure platform limitation, not a code gap). It keeps the `langgraph` orchestrator + app-side `pgvector` retrieval. Selecting `agent_framework` in pgvector mode is already rejected at request time with `ConfigResolutionError` → HTTP 409 (ADR 0022), never a silent fallback. **So "search done by Foundry IQ, not our code" is achievable in cosmosdb mode only; postgres keeps app-side retrieval by design.**
3. **The langgraph prompt fix is therefore NOT optional if postgres matters.** The `langgraph` path injects the same strict `compose_cwyd_instructions(CWYD_DEFAULT_BODY)` guardrail that makes `gpt-5.1` over-refuse (proven by the prompt-variant probes). Since postgres deployments are *locked* to `langgraph`, that over-refusal is a live defect for every postgres deployment — fixing the langgraph prompt is required for postgres mode to answer at all, not cleanup.

**Theoretical "Foundry IQ for postgres too" option (not supported, not recommended):** run postgres for chat-history but Azure AI Search for vectors + Foundry IQ KB. Blocked today by the `db_type ↔ index_store` coupling in `_enforce_mode_consistency`, and it would force paying for *both* postgres and Azure AI Search — defeating the postgres-only cost rationale. Out of scope unless the operator explicitly wants a hybrid.

**Net:** the operator's Foundry IQ decision needs **no new postgres design** — the split already exists (orchestrator-level, ADR 0021/0022). Remaining work is (a) wire the three cosmosdb Foundry IQ gaps; (b) fix the `langgraph` strict-prompt over-refusal so the postgres path (and any non-Foundry fallback) answers. The `providers/grounding/` registry idea is an *optional* refinement **inside** `agent_framework` (kb_mcp ↔ file_search) and is orthogonal to the postgres question.

## `langgraph` returns the out-of-domain fallback — root cause confirmed, fix pending operator decision (BUG-0028)

Operator: *"I switched from agent_framework to langgraph and I didn't get response — the frontend shows 'The requested information is not available in the retrieved data. Please try another query or topic.'"* This is the **known langgraph strict-prompt over-refusal** already tracked above as "Phase 3 (prompt softening)". Spent this session confirming it with live prompt-variant probes against the deployed model + index (read-only — no production code changed), and **filed [BUG-0028](../bugs.md#bug-0028--langgraph-over-refuses-in-domain-queries-strict-cwyd-guardrail--gpt-51-fallback)**.

### Confirmed diagnosis

Retrieval and message structure are **not** the cause — they are healthy:

* Direct `AzureSearch` retrieval for "tell me about employee benefits" returns **5 relevant hits** (`Benefit_Options.pdf`, `employee_handbook.pdf`) under both semantic and keyword search.
* `format_sources_block` injects clean `[docN]: <content>` lines; `_to_openai_messages` sends all messages including the sources.
* A **minimal** prompt + the same injected sources **grounds reliably** (probe A/B), so the model *can* answer.

The cause is the **shared strict CWYD prompt**. The `langgraph` path does raw chat completion and obeys the prompt literally; the `CWYD_GUARDRAIL` grounding clause ("if the retrieved documents do not contain enough information … your only response is [fallback]") makes `gpt-5.1` read "enough information" ultra-conservatively and refuse. `agent_framework` survives the identical prompt because Foundry IQ runs retrieval server-side and presents KB results as authoritative tool outputs.

Five guardrail-variant probes (each multi-run, temp 0.0 + default):

| Probe | "employee benefits" (broad) | other in-domain | out-of-domain | creative ("poem about dragons") |
|---|---|---|---|---|
| Strict (current) | REFUSED 8/8 | — | REFUSE ✓ | — |
| SOFT (permissive guardrail) | GROUNDED ✓ | GROUNDED ✓ | REFUSE ✓ | **LEAKS** ✗ (regresses BUG-0011) |
| SOFT2 (relevance-based) | REFUSED ✗ | GROUNDED ✓ | REFUSE ✓ | LEAKS ✗ |
| SOFT3 (+ explicit creative ban, strict body) | REFUSED ✗ | GROUNDED ✓ | REFUSE ✓ | REFUSE ✓ |
| SOFT4 (soft body + SOFT3 guardrail) | REFUSED 3/3 ✗ | GROUNDED ✓ | REFUSE ✓ | REFUSE ✓ |
| SOFT5 (SOFT4, "general essays" dropped from ban) | **boundary**: REFUSED, REFUSED, GROUNDED | GROUNDED ✓ ("company", "health plans") | REFUSE ✓ | REFUSE ✓ |

**Conclusions:** (1) temperature is not the lever (strict refuses 8/8 at temp 0.0 and default); (2) there is a genuine tension — a guardrail permissive enough to ground the broad "tell me about employee benefits" overview also tends to leak ungrounded creative content, and a guardrail strict enough to block creative content tends to refuse the broad overview; (3) SOFT5 is the best yet (grounds "company"/"health plans"/"handbook", refuses all OOD + creative) but the single broad "tell me about employee benefits" query sits **exactly on gpt-5.1's grounding decision boundary** for this small corpus (non-deterministic even at temp 0.0).

**Secondary finding (separate code defect):** `LLMProvider.complete()` calls `chat()` without threading `temperature` / `max_tokens`, so the `langgraph` path ignores the configured `openai_temperature=0.0` — the model runs at its default temperature, which contributes to the boundary non-determinism. Candidate for a small standalone fix.
