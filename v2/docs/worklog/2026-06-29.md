## Summary

Deleted the `auth_enforced` health signal and the client-side `AuthBlocked` "Authentication Not Configured" screen so the CWYD v2 frontend resolves identity purely from Azure Easy Auth `/.auth/me` — a real `userId` when a principal is present, otherwise the `DEFAULT_USER_ID` guest user. The SPA no longer renders a blocking wall. `require_admin_auth` stays as the orthogonal opt-in server-side admin-write gate (unchanged). Recorded as [BUG-0091](../bugs.md).

## Planned

- Remove `auth_enforced` from the `HealthResponse` model and its producer (Phase 1, backend).
- Strip the enforced/blocked concept from the SPA — `AuthPhase.Blocked`, `AuthState.authEnforced`, the `readAuthEnforced` helper, and the `AuthBlocked` component — so the bootstrap resolves identity from `/.auth/me` alone (Phase 2, frontend).
- Sync the tracked documentation: append one new sequential `BUG-####` row, annotate the superseded rows of the BUG-0046 identity plan, and write this worklog (Phase 3, docs).

## Bugs

- [BUG-0091](../bugs.md) (fixed) — removed the `auth_enforced` health signal and the `AuthBlocked` screen; the SPA resolves identity transparently from `/.auth/me`. Root cause: `auth_enforced` (a frontend-only UI flag added under BUG-0046) rendered a dead-end wall on an open / partially-wired deploy (cf. BUG-0089, BUG-0090) instead of degrading to the guest user.
- [BUG-0092](../bugs.md) (open) — found during Phase 4 cloud verification: the chat **History panel fails to load** on the split-host cloud deploy (`Could not load history: Unexpected token '<', "<!doctype "... is not valid JSON`). Root cause: `HistoryPanel.tsx` builds its request URL from the build-time `VITE_BACKEND_URL` (empty in the deployed App Service build) instead of the runtime `getBackendUrl()` `/config` seam, so history requests hit the SPA's own App Service origin and receive `index.html`. **Unrelated to the `auth_enforced` removal** — chat streaming and the health probe (which use the runtime seam) work.

## Decisions

- **PD-01 → Scenario A:** `require_admin_auth` (`AZURE_REQUIRE_ADMIN_AUTH`), `get_user_id`, and `requires_role` are left unchanged. Admin write-gating stays server-side and orthogonal to the SPA's identity resolution; only the frontend "enforced/blocked" UI concept was removed.
- **Clean-as-you-go (IN-01):** Phase 2 caught and removed an in-flight `readAuthEnforced` duplication while stripping `App.tsx`; no residual dead code remains.
- Recorded the change as a new sequential defect row (BUG-0091) rather than amending an existing one — none of the recent rows capture the removal/transparent-identity decision (per the defect-registry rule).
