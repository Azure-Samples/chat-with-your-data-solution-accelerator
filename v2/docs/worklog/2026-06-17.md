## Summary

Stood up a **new PostgreSQL-backed cloud environment** for local testing against cloud data — the first time the postgres deploy path has ever been provisioned (all prior cloud QA was cosmos). A planned **Phase A infra-hardening back-port** (durable Bicep for the ACR ARM-audience policy, the backend Container App `registries:` binding, and a frontend → Container App move) was implemented unit-by-unit and then **fully reverted to the committed baseline** per user direction: the frontend stays **App Service**, and the deploy must differ from baseline by **only** `databaseType=postgresql` (everything else default). The clean `azd up` then surfaced two latent, postgres-path-only deploy blockers — an out-of-range private-DNS-zone index ([BUG-0060](../bugs.md)) and an Event Grid delivery-identity authorization race ([BUG-0061](../bugs.md)) — both fixed, plus a corrupted local frontend `npm` install that was repaired. All infrastructure provisioned and all three services deployed; the deploy step then surfaced three more reverted-Phase-A / postgres-path gaps fixed live (ACR ARM-audience policy, backend ACR `registries:` binding, Function App storage firewall — [BUG-0062](../bugs.md)) plus a backend postgres connect-username mismatch ([BUG-0063](../bugs.md)), and `GET /api/health` now returns `pass`.

## Done

* **Reverted the Phase A infra-hardening back-port to baseline (per user direction).** Phase A had landed three units earlier in the session — A1 (ACR `azureADAuthenticationAsArmPolicy`), A2 (backend Container App `registries:` binding), and A3 (frontend re-scoped from App Service to a Container App). The user then directed that the **frontend stay App Service** and that the deploy be **"the same"** as baseline except for the postgres database type. `git restore`d `main.bicep`, `azure.yaml`, and `tests/infra/test_main_bicep.py` to the committed baseline (frontend back to `app-frontend-<SUFFIX>` App Service + `asp-<SUFFIX>` plan; `azure.yaml` `frontend.host: appservice`). Removed the stray `build-errors.txt`. The only structural `main.bicep` change carried into the deploy is the BUG-0060 DNS-zone-index fix below.
* **Created + configured the postgres azd environment.** New env; `AZURE_ENV_DATABASE_TYPE=postgresql`; main `location` set to a `location`-@allowed region; `AZURE_ENV_AI_SERVICE_LOCATION` set to a separate AI-allowed region; the four WAF flags (`ENABLE_MONITORING` / `SCALABILITY` / `REDUNDANCY` / `PRIVATE_NETWORKING`) all `false`; `AZURE_ENV_POSTGRES_ADMIN_PRINCIPAL_NAME` = the signed-in user (required, no default).
* **Resolved the region constraint.** The main `location` param's `@allowed` set is `{ australiaeast, eastus2, japaneast, uksouth }`, which does **not** include the chosen AI-services region — `azureAiServiceLocation` has a broader `@allowed` list that does. So main `location` and `azureAiServiceLocation` are deliberately set to **different** regions; an initial attempt to point the main `location` at the AI region failed param validation (the empty resource group from that attempt was deleted before re-running).
* **[BUG-0060] Fixed the out-of-range private-DNS-zone index — the only structural main.bicep change this session.** The first postgresql `azd up` failed ARM validation with `InvalidTemplate: The template reference 'avmPrivateDnsZones[7]' is not valid`. In postgresql mode the private-DNS-zone array has 7 entries (indices 0–6); the cosmos-only `aiSearch` module's private-endpoint DNS reference used the literal constant `dnsZoneIndex.search` (= 7), and ARM validates copy-index references **statically**, before the module's `if (databaseType == 'cosmosdb')` gate. Fixed by making the index mode-conditional — `avmPrivateDnsZones[databaseType == 'cosmosdb' ? dnsZoneIndex.search : dnsZoneIndex.postgres]` — so it is never the literal `7` in postgres mode (and resolves to an unused in-range index there). No behavior change for cosmos. `az bicep build` clean; provision then ran.
* **Repaired a corrupted local frontend npm install (environment issue, not a code defect).** During `azd package`, the frontend build reported 13 TypeScript errors (`App.tsx`, `AdminLayout.tsx`, `IngestData.tsx`) — all from `react-router-dom@7` shipping with its `dist/**/*.d.ts` / `.d.mts` type files **missing** from `node_modules`. No frontend source was touched: a clean `Remove-Item -Recurse -Force node_modules; npm install` at the v2 root restored the type files and `npm run build` passed (vite built `dist`). No `bugs.md` entry (local environment corruption, not a tracked-code defect).
* **[BUG-0061] Broke the Event Grid delivery-identity deploy-time chicken-and-egg.** Provision reached the backend Container App, the Function App, the frontend App Service, and the PostgreSQL Flexible Server, then failed with `Managed identity for event subscription does not have authorization to deliver to the endpoint` (the `blob-created-to-doc-processing` subscription → `doc-processing` queue). The AVM system-topic module creates the topic, its system-assigned MI, **and** the event subscription in one nested deployment, while the role assignment that authorizes the MI is external and ordered **after** the module — so the subscription's `deliveryWithResourceIdentity` preflight has no role yet. Unblocked by pre-creating the topic MI's `Storage Queue Data Message Sender` role on the storage account **under bicep's exact deterministic `guid(storageAccount.id, eventGridSystemTopicName, roleId)` name** (via a throwaway single-resource Bicep deployment), so the subscription preflight finds the role *and* bicep's own `eventGridQueueSenderRole` PUT is an idempotent no-op. (An earlier attempt granted the role under a random GUID name, which unblocked the subscription but then collided by-content with bicep's deterministic-named role on the next provision — `RoleAssignmentExists`; the deterministic-name grant avoids that.)
* **Infra provisioned (provision step fully green).** Container Apps Environment, both App Service plans (frontend + functions), Storage, AI Services (speech + content safety), Container Registry, Foundry + project + the three model deployments (gpt-5.1 / o4-mini / text-embedding-3-large), the PostgreSQL Flexible Server, the backend Container App, the Function App, and the frontend App Service all created. The azd `post-provision` hook connected to the pg server as the signed-in admin and reported `pgvector extension ready` (search-index / knowledge-base steps skipped — pgvector mode, no `AZURE_AI_SEARCH_ENDPOINT`). azd outputs emitted (backend Container App FQDN, frontend `azurewebsites.net`, function URL, pg host, AI endpoints).
* **Enabled the ACR ARM-audience auth policy (live patch — known reverted-A1 gap).** With provision green, the `publish-backend` step failed `401 token validation failed` on `POST .../oauth2/exchange` (and `admin user is not enabled`): the AVM container-registry module ships `azureADAuthenticationAsArmPolicy` **disabled**, so azd's Entra ARM-audience token can't be exchanged for an ACR refresh token to push the backend image. This is exactly the gap Phase A unit A1 had fixed in Bicep before the revert — so it was applied **live** instead: `az acr config authentication-as-arm update --registry cr<SUFFIX> --status enabled` (admin user left **off** — the ARM-audience path is the secure fix). Re-ran `azd deploy`; the backend image push then authenticated and proceeded. Tracked as the open `ACR-AAD-AS-ARM-BICEP-DEBT` row in [development_plan.md](../development_plan.md) §0.1 (durable Bicep fix deferred with the rest of the reverted Phase A).
* **Bound the backend Container App to ACR with the UAMI (live patch — known reverted-A2 gap).** After the ACR policy fix, the backend image push succeeded but the new revision failed to pull the image: `UNAUTHORIZED: authentication required` (the Container App had no `registries:` binding, so it attempted an anonymous pull). This is exactly the gap Phase A unit A2 had fixed in Bicep before the revert — applied **live** instead: `az containerapp registry set -n ca-backend-<SUFFIX> -g <RESOURCE_GROUP> --server cr<SUFFIX>.azurecr.io --identity <UAMI resource id>`. Re-ran `azd deploy`; backend + frontend reported **Done**. Tracked as the open `BACKEND-CA-ACR-REGISTRIES-BICEP-DEBT` row in [development_plan.md](../development_plan.md) §0.1.
* **[BUG-0062] Opened the Function App storage firewall for the Flex Consumption platform (live patch).** The function deploy then failed `InaccessibleStorageException` / `403` against `st<SUFFIX>`: the storage account ships `networkRuleSet.defaultAction=Deny` + `bypass=AzureServices` with no resource-instance rule, and `allowSharedKeyAccess=false` (policy) leaves no key fallback, so with `enablePrivateNetworking=false` the Flex Consumption platform cannot reach storage at deploy or runtime. Fixed live: `az storage account update -n st<SUFFIX> -g <RESOURCE_GROUP> --default-action Allow` (data still Entra/RBAC-protected; only the network-ACL default flips). Granting the deployer blob RBAC + adding the deployer IP did not help (the blocked principal is the platform); the `azd 1.25.5 → 1.25.6` update was a red herring. The function then deployed in ~3 min. Durable bicep fix pending — see [BUG-0062](../bugs.md).
* **All three services deployed.** `azd deploy` reported `SUCCESS` for backend, frontend, and function (`app-frontend-<SUFFIX>.azurewebsites.net`, `func-<SUFFIX>.azurewebsites.net`, and the backend Container App FQDN).
* **[BUG-0063] Fixed the backend postgres connect-username mismatch (live patch) — backend now healthy.** With all services deployed, `/api/health` timed out: the backend's azd revision was in `ActivationFailed` (0 replicas). The FastAPI lifespan opens the asyncpg pool eagerly, authenticating with the **UAMI** token but connecting with username `AZURE_POSTGRES_ADMIN_PRINCIPAL_NAME` = the **deployer** UPN — which [main.bicep](../infra/main.bicep) wires for the backend (~L1760) and function (~L2066) to `postgresAdminPrincipalName`. Entra-only postgres rejects the login (token subject `id-<SUFFIX>` ≠ the deployer role), so the pool never opens and the lifespan raises before `yield`. Fixed live: `az containerapp update -n ca-backend-<SUFFIX> -g <RESOURCE_GROUP> --set-env-vars AZURE_POSTGRES_ADMIN_PRINCIPAL_NAME=id-<SUFFIX>` (the UAMI is already a postgres AAD admin under `id-${solutionSuffix}`). The new revision boots and `GET /api/health` → `{"status":"pass", database=pass (db_type=postgresql), foundry_iq=pass, search=skip}`. Durable bicep fix pending — see [BUG-0063](../bugs.md).
* **[BUG-0064] Set the orchestrator the backend actually reads to `langgraph` (live patch) — chat now routable on pgvector.** With the backend healthy, the deployed env still selected the wrong orchestrator: [main.bicep](../infra/main.bicep) L1776 emits `ORCHESTRATOR=agent_framework`, but the backend reads `CWYD_ORCHESTRATOR_NAME` (`OrchestratorSettings` `env_prefix=CWYD_ORCHESTRATOR_`), so the bicep value is ignored and the orchestrator falls to its default `agent_framework` — which is incompatible with pgvector (no Foundry IQ KB) and would 409 at chat time per [ADR 0022](../adr/0022-config-resolution-error-on-incompatible-overrides.md). Health stayed green because the orchestrator is built per request, not in the lifespan. Fixed live: `az containerapp update -n ca-backend-<SUFFIX> -g <RESOURCE_GROUP> --set-env-vars CWYD_ORCHESTRATOR_NAME=langgraph` — new revision Running + Healthy, `/api/health` `pass`. Durable bicep fix pending — see [BUG-0064](../bugs.md).
* **Local-against-cloud run stood up + the langgraph + pgvector + cloud-Postgres path validated end-to-end.** Created the gitignored `v2/.env` from `azd env get-values` with two local overrides: `AZURE_UAMI_CLIENT_ID` left **blank** (so the credentials registry resolves `AzureCliCredential` from `az login`, not `ManagedIdentityCredential`), and `AZURE_POSTGRES_ADMIN_PRINCIPAL_NAME` kept as the **signed-in deployer UPN** — the local connect user and a pg Entra admin — **not** the cloud's `id-<SUFFIX>` UAMI principal (the inverse of the BUG-0063 cloud fix). `CWYD_ORCHESTRATOR_NAME=langgraph` (pgvector only path today). Opened a pg firewall rule for the dev workstation public IP on `psql-<SUFFIX>`, and granted the signed-in user the data-plane roles the local process needs (the deployment grants these to the UAMI, not the human): **Cognitive Services OpenAI User** + **Cognitive Services User** on `aisa-<SUFFIX>`, **Storage Blob Data Contributor** + **Storage Queue Data Contributor** on `st<SUFFIX>` (pg access is already covered — the deployer is the pg Entra admin). Ran the backend natively (`uv run --env-file .env uvicorn backend.app:app --app-dir src --port 8000`): the lifespan opened the asyncpg pool to cloud Postgres via the dev user's Entra token, `/api/health` + `/api/health/ready` → `{"status":"pass", database=pass (postgresql), foundry_iq=pass, search=skip (pgvector)}`; a `langgraph` `POST /api/conversation` → `200` with a full grounded answer + a new `conversation_id` (citations empty — no documents ingested into pgvector yet); `GET /api/history/conversations` → `200` (read-back from cloud Postgres). Confirms chat write + history read against cloud Postgres work locally. Frontend + Functions (local ingestion) and the cloud smoke matrix are the next steps; the `agent_framework`-on-pgvector feature is gated on the approach decision (A1 static RAG / A2 function tool / A3 Foundry IQ over an MCP knowledge source) + an ADR 0022 supersession.

## Decisions

* **Phase A reverted; deploy differs from baseline by only `databaseType=postgresql`.** Per the user: the frontend stays **App Service** and the deploy should be "the same" as baseline otherwise. The A1/A2/A3 infra-hardening back-ports were rolled back to the committed baseline; the only structural main.bicep change carried into the deploy is the BUG-0060 DNS-zone-index fix (without it, ARM rejects the template in postgres mode — unavoidable).
* **Full `azd up` (not just provision), env in a UK region, postgres admin = signed-in user.** Confirmed with the user.
* **Two-region split is intentional.** The main `location` and `azureAiServiceLocation` are different regions because the main `location` `@allowed` list excludes the chosen AI-services region.
* **Deploy blockers fixed forward, recorded in `bugs.md`, durable bicep back-ports deferred.** BUG-0060 (DNS index) was fixed in `main.bicep` because ARM validation is unavoidable. BUG-0061 (Event Grid role race) was unblocked with a **live cloud role grant under bicep's deterministic name** rather than a bicep edit, matching the established pattern of pausing bicep edits and logging a "durable back-port pending" note. The frontend type-file corruption was a local environment repair, not a code change.
