Confirmed live against the service: the sample DOCX returned `pages=1`, per-page `lines=[0]`, `paragraphs=848`, `content=29335`; a sample PDF returned `pages=4`, `lines=[4,6,34,25]`. Fixed by adding a paragraph fallback: when the page pass yields zero chunks, emit one `Chunk` per `result.paragraphs` entry (stripping empties, keeping `index` dense), mirroring `TextParser`'s per-paragraph chunking. The fallback fires only when the page pass produced nothing, so PDFs / images keep their one-chunk-per-page shape and never double-emit. Three new tests (paragraph fallback when pages are empty; pages-with-text suppress the fallback / no duplication; empty paragraphs skipped with dense indices) — the two behavior tests failed pre-fix; full `test_document_intelligence_parser.py` `25 passed`, `pyright` `0/0/0` on the parser, shared gates `237 passed`. Verified live with the real parser over the sample files: PDF → 4 chunks (unchanged), DOCX → 848 chunks. No infra / model / frontend change.

Deliberately **not** a raise: a legitimately text-free input also yields zero chunks and raising would poison-loop it forever — the handler cannot distinguish a parser failure from a file with no extractable text. The existing `test_zero_chunks_short_circuits_embed_and_search` was tightened to capture at `WARNING` and assert `levelname == "WARNING"` (failed pre-fix); `test_handler.py` `3 passed`, `pyright` `0/0/0`, process-narrative gate `117 passed`.

## Decisions

* **Traced through the `_execute` seam rather than waiting on the live queue.** The restarted host was not draining the main queue during the trace (a local-dev listener artifact after the overnight crash), so a queued message sat undrained. Running the single monkeypatch seam `_execute` in-process gives the real exception with a full traceback immediately and exercises the exact production orchestration — a faster, deterministic root-cause path than fighting the queue-trigger backoff.
* **BUG-0034 stays open.** The host config is incidentally correct now, but the proper code fix (fail-fast validation) and the `v2/.env` backfill have not landed; those are separate test-first units. The trace was a diagnosis, not a fix.
* **Honest error at the boundary the operator sees, scoped precisely.** The false-green is fixed where the operator experiences it — the synchronous `POST /api/admin/documents` response — not only in the async Functions tier (unit a). Chose a registry-identity probe (`registry.get(ext) is registry.get("pdf")`) over a hardcoded `{"pdf","docx"}` set (would violate Hard Rule #4 and silently miss any future DI-routed extension) and over a blunt block-all-uploads-when-endpoint-missing (would itself be "incorrect logic" — it would reject txt / md / html that parse locally and need no endpoint). Kept the guard in the router (config-presence guards return `HTTPException` directly there) rather than introducing a parser-capability protocol (a structural, multi-unit change — deferred, ask-first).
* **Queue-name drift logged inside BUG-0034, not a new bug.** It is a second cause of the same observed symptom and is a local-dev config drift, not a shipped defect, so it folds into the existing BUG-0034 entry rather than opening BUG-0035.
* **BUG-0047 is a backend fix, not a frontend one.** The user's instinct ("the header is missing from the frontend") is half-right — the *claims* header is what the gate needs, but that blob is the server-side trust boundary for admin RBAC and must never be forged or forwarded by the browser (a browser-supplied claims blob would let anyone assert the admin role; this is the documented BUG-0046 Security note). The frontend's modular `userIdHeaders()` seam already forwards everything it legitimately can (the partition id). So the one correct, modular fix is the single backend conditional in `requires_role._checker`, which fixes all admin endpoints simultaneously — not a per-call frontend header.
* **Upload `503` was operational, not a code defect — the BUG-0034 guard worked as designed.** A live `POST /api/admin/documents` returned the `503` "Document parsing is not configured" guard because the running backend had been launched **without** `--env-file .env`. Pydantic-settings sub-`BaseSettings` models don't inherit `env_file`, so `services_endpoint` was empty in the cached `get_settings()` singleton, and `--reload` never re-reads `.env`. Stopped the stale process and relaunched with `uv run --env-file .env uvicorn backend.app:app --app-dir src --host 127.0.0.1 --port 8000 --reload`; settings then resolved `services_endpoint` to an `https` URL (`is_https True`, `len 52`) and `/api/health` + `/api/admin/status` returned `200`, so the upload guard passes. Conclusion: the guard is correct (it surfaced a genuinely unindexable config); the fix is to start the backend with the env file, not to change the guard. No new bug opened. (Also noted: a transient git merge-conflict marker had appeared in `services/conversation.py` mid-session causing a `SyntaxError`; a grep over `v2/src` confirmed no markers remain.)
* **Delete keeps the source-name contract; the fix needs no server-side filter, schema change, or API change.** The user's instinct ("use the document id instead of the name") is directionally right — deletion must act on chunk ids — but the delete *already* deletes by id; only the id-*discovery* step failed. The admin documents list surfaces *sources* (one logical document = N chunks, each with its own id), so the frontend has no per-chunk id to send and the `DELETE /…/{source}` route is correct. The minimal, modular fix mirrors the proven `list_sources` workaround already in the same provider (page + match `title` client-side) rather than (a) changing the index schema to make `title` filterable — a field drop/recreate + full reindex, heavyweight and structural — or (b) changing the API to take ids, which the frontend cannot feed. Matching client-side additionally eliminates the OData-injection seam the previous `title eq '<source>'` filter opened on a user-supplied filename.
* **DOCX fix reuses Document Intelligence's own `paragraphs`, not a new chunker or a single giant chunk.** v2 deliberately removed the chunker primitive (parsers own chunking per format) and `batch_push` embeds each `Chunk.content` directly with no downstream re-chunking, so the parser must bound chunk size itself. Three options were on the table: (a) emit one chunk from the flat `result.content` — rejected: a single ~29 KB chunk strains the embedding token ceiling and gives one undifferentiated vector with poor retrieval; (b) re-split `result.content` with a regex like `TextParser` — rejected: re-implements splitting the service already did; (c) emit one chunk per `result.paragraphs` entry — chosen: Document Intelligence already returns the DOCX pre-split into 848 semantic paragraphs, which is exactly the unit `TextParser` uses, so it bounds size, matches the codebase convention, and invents no abstraction. Gating the fallback on "the page pass produced zero chunks" (not on a format/extension check) keeps it format-agnostic and guarantees PDFs — which return both `pages[].lines` and `paragraphs` — never double-emit. No infra, model, index, or frontend change; the `prebuilt-layout` model and DI endpoint are unchanged.
* **"Still showed success" is the async contract, not a second bug — hardened visibility instead of changing the contract.** The upload `200` means "accepted + enqueued," because parse / embed / index run asynchronously in the Functions `batch_push` consumer after the HTTP response has already returned; the synchronous boundary structurally cannot report the parse outcome, and the BUG-0034 guard already does the only sensible synchronous pre-flight (refuse a type the deployment can't index at all). The real gap was that a downstream zero-chunk outcome was logged at `info` and otherwise swallowed. Chose the minimal in-scope fix — elevate that log to `warning` so it is visible in telemetry — over the two heavier alternatives: (a) raising on zero chunks (rejected — poison-loops legitimately text-free files), and (b) a per-document processing-status store the admin UI could display (a real improvement, but structural / multi-unit — deferred to its own scoped piece, not done inline). The user picked the log-hardening option.

### BUG-0051 — Admin document upload 503s (backend container missing storage env vars)

With the Admin button now rendering, QA tried to upload a document and hit `503 {"detail":"Document storage is not configured for this deployment."}` ("I thought you configured all the variables").

**Diagnosis (live).** The backend admin endpoints ([admin.py](../../src/backend/routers/admin.py)) gate upload / URL ingest / reprocess on `settings.storage.documents_container` **and** `settings.storage.doc_processing_queue` being non-empty, and `GET /api/files/{filename}` ([files.py](../../src/backend/services/files.py)) needs the documents container; `StorageSettings` ([settings.py](../../src/backend/core/settings.py), `env_prefix="AZURE_"`) defaults every storage field to `""`. `az containerapp show` on the backend listed **none** of `AZURE_STORAGE_ACCOUNT_NAME` / `AZURE_DOCUMENTS_CONTAINER` / `AZURE_DOC_PROCESSING_QUEUE` in its container `env`. Root cause: the `backendContainerApp` module in [main.bicep](../../infra/main.bicep) never wired the storage vars — they are set **only** on the function app block (and emitted as stack outputs) — so the deployed backend resolved all three to empty and the gate tripped. Filed as **BUG-0051** (infra, high). Not an RBAC gap: the shared UAMI already holds Storage Blob Data Owner + Storage Blob Data Contributor + Storage Queue Data Contributor + Storage Queue Data Message Sender on the account (verified with `az role assignment list` scoped to the storage account; the function app does ingestion under the same identity). The blob/queue endpoints are derived from the account name by `resolve_storage_endpoints` ([storage_endpoints.py](../../src/functions/core/storage_endpoints.py)), so only the account name + container + queue are needed (no explicit `AZURE_STORAGE_BLOB_ENDPOINT`), matching the function app.

**Live hotfix (QA unblocked).** `az containerapp update -g <RESOURCE_GROUP> -n ca-backend-<SUFFIX> --set-env-vars AZURE_STORAGE_ACCOUNT_NAME=<account> AZURE_DOCUMENTS_CONTAINER=documents AZURE_DOC_PROCESSING_QUEUE=doc-processing` (additive, reversible) → new revision `Healthy` + `Running`, env vars confirmed present. **Verified end-to-end:** a `.txt` smoke upload to `POST /api/admin/documents` returned `200` `{"filename":"cwyd-qa-smoke.txt","blob_path":"documents/cwyd-qa-smoke.txt","ingestion_job_id":"…","queued":true}` — proving the config gate passes, the blob write succeeds, and the queue enqueue succeeds, all under the UAMI. The smoke blob was deleted afterward (`az storage blob delete --auth-mode login`; the not-yet-indexed chunk delete returned `404`, as expected).

**Durable fix.** Added the same three storage env entries the function app uses (`AZURE_STORAGE_ACCOUNT_NAME` = `effectiveStorageName`, `AZURE_DOCUMENTS_CONTAINER` = `documentsContainerName`, `AZURE_DOC_PROCESSING_QUEUE` = `docProcessingQueueName`) to the `backendContainerApp` container `env` array in `main.bicep`, after the `ORCHESTRATOR` entry, so the wiring survives re-provision. `az bicep build` → exit 0, no new errors. This is a config addition to an existing resource block (not structural). No app-code, RBAC, or frontend change.

**Lesson.** The backend Container App needs the **same** storage wiring as the function app whenever it owns a write/enqueue surface (admin upload, URL ingest, reprocess) — the two runtimes share one container + queue. The bicep gap meant only the ingestion runtime could talk to storage; the API runtime that *accepts* uploads could not.
