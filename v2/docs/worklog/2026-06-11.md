## Decisions

* Post-answering is not a defect. Research confirmed v1 ships a full post-answering feature (a default prompt, a fact-checking validator, and an operator filter message), so the v2 post-answering trio matches v1 and stays as is.
* Bringing v1 prompts into v2 is scoped to the default prompt text only. The v1 per-business-unit assistant-type system (Default, Contract assistant, Employee assistant) is out of scope for v2.
* All six items were recorded in [bugs.md](../bugs.md) rather than split across the development plan debt queues, by request. No `BUG-####` pointer rows were added to the development plan this turn; they can be added later if a defect needs phase-audit visibility.
* `BUG-0003`'s recorded root cause was corrected after reproducing it locally against the live cloud. The `503` is not caused by an unset search endpoint — it fires with search fully configured, because `list_sources` requests a facet on the non-facetable `title` field. The original empty-endpoint hypothesis is a separate latent pass-through path.
* `BUG-0011` was settled with the operator as guardrail composition (Decision A — extract a single reused `CWYD_GUARDRAIL` constant; Decision Z — wrap the body on both sides, prefix and suffix; **Z later superseded the same day** — deduped the body against the guardrail and switched to appending the guardrail once, at the end, for last-word precedence and zero duplication). Two corrections to the recorded root cause: (a) RAI validation of the authored `cwyd_agent_instructions` on save was **already in place** (`PROMPT_FIELDS` + frontend `RAI_GUARDED_FIELDS`), so the stale note was corrected; (b) RAI-classifying end-user chat input (beyond the existing `ContentSafetyGuard`) is a separate, larger decision — a Foundry round-trip per chat turn — and is intentionally **out of scope** for this fix, not bundled.
* `BUG-0020` (the live-run `401`) was fixed out of numerical order because it is a blocker for the entire `agent_framework` chat path (the default Foundry IQ deployment) and surfaced only on the first end-to-end live run. The fix authenticates the MCP connect at the connection level (default `Authorization` header on a caller-owned httpx client) and retains the per-call `header_provider` hook; it was cross-checked against MACAE, whose one bearer-attached MCP client sets the header as a client default. The open chat-UI defects (`BUG-0013`–`BUG-0019`) are unaffected and remain queued.
* Config-persistence "reset to default" was settled as **Option A** (front-end clears every override with an all-`null` PATCH) over **Option B** (a new backend `DELETE /api/admin/config` endpoint). Option A reuses the already-tested PATCH route + RFC 7396 clear semantics for zero new backend surface; sending the prompt fields as `null` (rather than empty strings) also skips the per-field Foundry RAI round-trip the PATCH route runs on non-null prompt values, so reset is cheap. Option B would add an endpoint, a router test, and an audit-row path for no behavioral gain.
* Flagged a latent gap **not** fixed inline and **not** yet filed as a bug (pending operator decision): `openai_temperature` and `openai_max_tokens` are persisted and shown in the admin form, but may not be threaded into the chat call — the `foundry_iq` LLM provider is built once at startup from env. Candidate `BUG-0022` if the operator wants it tracked — re-verify first, since the `agent_framework` orchestrator does pass `temperature` / `max_tokens` via `ChatOptions`, so confirm whether the gap still exists. (`BUG-0021` was assigned to the agent-not-found chat blocker above.)
