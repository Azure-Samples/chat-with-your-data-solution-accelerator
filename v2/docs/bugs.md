## Purpose

This file is the canonical, file-based registry of **defects** in CWYD v2: behavior that is wrong, broken, or regressed against intent. It exists so bug history survives across sessions and is never trapped in ephemeral agent memory.

Record a bug here the moment it is observed, even before it is fixed. Update the same entry when it is fixed. Every entry is self-contained: a reader should understand the symptom, the root cause, and the fix without opening any other file.

## Relationship to the development plan

[development_plan.md](development_plan.md) §0.1 (backend) and §0.2 (frontend) are the **phase debt and task queues**: deferred work, refactors, and follow-ups that clear on each phase's end-of-phase audit cadence (Hard Rule #12). They are not a defect log.

The split:

* A **defect** (wrong or broken behavior) is recorded here in `bugs.md`. This is the canonical source of truth for defects.
* A **debt or task item** (deferred work, planned refactor, scoped follow-up) stays in the development plan debt queues.

When a defect also needs phase-audit visibility, the development plan keeps a one-line pointer row that references the `BUG-####` id here, and the full detail lives in this file. There is one source of truth per concern, never two.

## Conventions

### Identifiers

Bugs use a sequential, zero-padded id: `BUG-0001`, `BUG-0002`, and so on. Ids are never reused. The next free id is one greater than the highest id already in the registry.

### Closed-set fields

The registry uses three closed sets. Use only these values.

Area: `backend`, `frontend`, `infra`, `functions`, `docs`, `ci`.

Severity: `blocker` (nothing ships or a pipeline is dead), `high` (a core path is broken), `medium` (a path is degraded with a workaround), `low` (cosmetic or minor).

Status: `open`, `in-progress`, `fixed`, `wontfix`, `duplicate`.

### Placeholder rule

This file is tracked and may reach public GitHub. Never write real environment values (subscription, tenant, resource group, azd env name, resource suffix, identity ids, real FQDNs). Use the placeholder tokens defined in [adr/0019-no-env-specific-content-in-tracked-files.md](adr/0019-no-env-specific-content-in-tracked-files.md), for example `<SUFFIX>`, `<RESOURCE_GROUP>`, `<AZURE_SUBSCRIPTION_ID>`. Environment-variable names such as `AZURE_AI_SERVICES_ENDPOINT` are generic and may be written verbatim.

### How to add a bug

1. Allocate the next `BUG-####` id.
2. Add a row to the Registry table with status `open` (or `in-progress` if already being worked).
3. Add a `### BUG-####` subsection under Details with the symptom, root cause, fix, and references.
4. Cross-reference the day's worklog entry under [worklog/](worklog/), and add a pointer row in the development plan debt queue only if the defect needs phase-audit visibility.

## Details

### BUG-0001 — Embeddings routed to the project endpoint and missing `dimensions`

Area: backend. Severity: blocker. Status: fixed (found and fixed 2026-06-10).

Symptom: the `batch_push` ingestion pipeline could not produce a single embedded chunk. `FoundryIQ.embed()` returned `404`.

Root cause: `FoundryIQ.embed()` reused the project-scoped OpenAI client that is correct for chat and agents. The Foundry **project** route exposes no `embeddings` path, so every embeddings request returned `404`. The call also never passed `dimensions`, so a `text-embedding-3-large` deployment would emit 3072-dimension vectors against the 1536-dimension `content_vector` index field.

Fix: a new `FoundryIQ._get_embeddings_client()` targets the **account** endpoint (`AZURE_AI_SERVICES_ENDPOINT` plus `/openai/v1`), and `embed()` now passes `dimensions=settings.openai.embedding_dimensions`.

Why it was not caught earlier: `test_foundry_iq.py` fully mocks the OpenAI client, which hid the project-versus-account routing distinction.

References: [worklog/2026-06-10.md](worklog/2026-06-10.md); [development_plan.md](development_plan.md) §0.1 `INGEST-EMBED-DOCKEY`.

### BUG-0002 — Parser chunk id is an illegal Azure AI Search document key

Area: backend. Severity: blocker. Status: fixed (found and fixed 2026-06-10).

Symptom: after embeddings were fixed, the Azure AI Search push step failed with `InvalidDocumentKey`, so no document reached the index.

Root cause: parsers minted `Chunk.id` as `f"{source}__{index}"`. When `source` is a filename, its extension dot makes the key illegal. Azure AI Search document keys allow only letters, digits, `_`, `-`, and `=`.

Fix: a new Stable Core helper `BaseParser.make_chunk_id(source, index)` hashes the readable `f"{source}__{index}"` through SHA-256 into a key-safe hex digest (mirrors the v1 `source_document.py` hashing precedent). Both parsers call it. The readable name survives on `Chunk.source` and the Search `title` field, and the read-side `_to_result` already treats `id` as opaque.

Why it was not caught earlier: the parser tests asserted the raw `id` literal, which baked the invalid key charset into the expectation. A new `test_base.py` (5 tests) now asserts key safety with a charset regex against the helper contract instead of a magic string.

References: [worklog/2026-06-10.md](worklog/2026-06-10.md); [development_plan.md](development_plan.md) §0.1 `INGEST-EMBED-DOCKEY`.

### BUG-0020 — KB MCP connect is unauthenticated (401 on chat)

Area: backend. Severity: high. Status: fixed (found and fixed 2026-06-11).

Symptom: sending a chat message on a Foundry IQ deployment failed with `Message failed ('Failed to enter context manager.', ExceptionGroup('unhandled errors in a TaskGroup', [HTTPStatusError("Client error '401 Unauthorized' for url 'https://srch-<SUFFIX>.search.windows.net/knowledgebases/cwyd-kb/mcp?api-version=2025-11-01-preview'")]))`.

Root cause: the `agent_framework` orchestrator (`backend/core/providers/orchestrators/agent_framework.py`) attached the Search data-plane bearer to the Knowledge Base MCP tool through the SDK's `header_provider` hook only. The Agent Framework SDK applies that hook exclusively inside `MCPStreamableHTTPTool.call_tool()` (it sets a `contextvars.ContextVar` that an httpx request event hook reads). The MCP session-initialize connect handshake runs earlier — when the tool's async context manager is entered (`MCPTool.__aenter__` → `connect()`), before any tool call — so the connect request carried no `Authorization` header. Azure AI Search rejected the unauthenticated connect with `401`, and the SDK re-raised it as `ToolExecutionException("Failed to enter context manager.")`. The `/health/ready` search probe stayed green because it exercises the normal data-plane index credential, a different surface from the KB MCP connect.

Fix: `run()` now constructs an `httpx.AsyncClient` carrying the same bearer as a default `Authorization` header and passes it to `_build_kb_tool(..., http_client=...)`, which forwards it to `MCPStreamableHTTPTool(http_client=...)`. httpx applies default headers to every request, so the connect handshake is authenticated as well as tool calls. The `header_provider` hook is retained (belt-and-suspenders for tool calls; the same bearer merges idempotently). Because the MCP streamable-HTTP transport never closes a caller-supplied client (`mcp.client.streamable_http.streamable_http_client` only manages clients it creates), `run()` owns the client lifecycle and closes it in a `finally` that covers both the streaming path and an agent-construction failure. Cross-checked against MACAE, whose one bearer-attached MCP client path sets `headers["Authorization"]` as a default header on the httpx client (`src/backend/v4/common/services/mcp_service.py`), confirming the connection-level default-header approach.

Why it was not caught earlier: this was the first live end-to-end exercise of the agent path against a real Search service; orchestrator unit tests fully fake the FoundryAgent and never enter the MCP tool's context manager, so the connect handshake was never exercised. New tests assert the KB tool is built with a default-`Authorization`-header client and that the client is closed after a normal run and after an agent-construction failure.

### BUG-0021 — `agent_framework` chat 404s: persistent-assistant vs name-referenced Foundry agent mismatch

Area: backend. Severity: blocker. Status: fixed (found 2026-06-11, fixed 2026-06-12).

Symptom: every `POST /api/conversation` on the default `agent_framework` orchestrator returns `500` (buffered) / an `error` SSE frame (streaming). Backend log: `ChatClientException: ... 404 - Agent 'cwyd' not found`. The failure surfaces in the chat-completion stream (`agent_framework_openai/_chat_client.py`), i.e. *after* `_prepare_run_context` — so it is downstream of, and distinct from, the BUG-0020 MCP-connect 401 (which is fixed).

Root cause: two different Foundry agent surfaces are crossed.

- Provisioning (`BaseAgentsProvider.get_or_create_agent` → `FoundryAgentsProvider`) calls `azure.ai.agents.aio.AgentsClient.create_agent(...)`, which creates an **Assistants-API persistent agent** identified by an `asst_` id. Confirmed live: the Foundry project holds exactly one agent — an `asst_`-prefixed id with `name='cwyd'` and `model='gpt-5.1'`, i.e. an Assistants-API persistent agent.
- Invocation (`AgentFrameworkOrchestrator` → `agent_framework_foundry.FoundryAgent(agent_name="cwyd", allow_preview=True)`) references the agent through the **Responses API by name** (`get_openai_client(agent_name=...)` / `agent_reference {"name": "cwyd"}`). That name-reference resolves a Foundry *named/prompt* agent resource, not an Assistants-API `asst_` persistent agent, so Foundry returns 404.
- The conversation router compounds the gap: it calls `get_or_create_agent(CWYD_AGENT, db)` but **discards the returned `asst_` id** and passes the friendly `CWYD_AGENT.name` ("cwyd") to the orchestrator.

Contributing context: the project endpoint was recently switched to a different Foundry account/project (per `.env`). In the new project only the persistent-assistant create path has run, so no name-referenceable agent resource exists.

Fix direction (recommended — structural, Hard Rule #10, needs operator consent): align the two halves of the decoupled-hybrid (dev_plan §0.1 `B1-MAF-MISLABEL` / `B-IMPL-AUDIT`) on the **name-referenced Prompt Agent** resource type. Switch `FoundryAgentsProvider`'s create path from `azure.ai.agents.AgentsClient.create_agent` (which mints an Assistants-API `asst_` agent) to the Foundry **Prompt Agent** path (`PromptAgentDefinition` + `create_version` on the projects SDK), so a name(+version)-resolvable `cwyd` agent exists for the orchestrator's `FoundryAgent(agent_name=..., agent_version=...)` invocation. This is exactly the alternative the dev_plan **B-Spike** flagged (`PromptAgentDefinition` + `create_version` vs `AgentsClient.create_agent`) but never switched.

Rejected alternatives: (a) invoke the existing `asst_` agent by id — the OSS `FoundryAgent` exposes no `agent_id` parameter (name + optional version only, for Prompt/Hosted agents) and no `agent_framework` persistent-agent (Azure AI Agent Service threads/runs) client is installed; (b) toggling `allow_preview` does not change the resource type the reference resolves — both code paths reference a Prompt/Hosted agent by name.

Why it was not caught earlier: `test_agent_framework.py` injects a fake agent factory, so the persistent-assistant-vs-name-reference distinction is never exercised against the live Responses API. The "Thinking" panel appears as soon as the SSE stream opens, so a screenshot showing "Thinking" does not prove the agent actually ran.

Resolution (2026-06-12, [ADR 0025](adr/0025-foundry-prompt-agent-ga-pattern.md)): fixed via the Foundry **Prompt Agent GA pattern** — exactly the recommended direction above. `FoundryAgentsProvider.get_or_create_agent` now builds a `PromptAgentDefinition` and calls `AIProjectClient.agents.create_version(agent_name=...)` (the `asst_` `AgentsClient.create_agent` path is removed), persisting and returning the agent **name**; a cold start validates the name with `agents.get` (`ResourceNotFoundError` → recreate) and a concurrent-create `409` re-reads and reuses the existing agent. Invocation no longer uses the by-name `FoundryAgent` Responses path: the orchestrator composes a client-side `agent_framework.Agent` over `FoundryChatClient` through the shared `build_agent` seam and drives it as an async context manager. KB grounding moved server-side (`MCPTool(project_connection_id=...)`), which also retired the BUG-0020 client-side bearer / `httpx` machinery. Both halves now reference one name-addressed Prompt Agent, so the resource-type mismatch cannot recur. Landed across units U1–U8 + U10 (RAI migrated to the same seam in U8); ADR 0025 carries the full decision record and the multi-agent coordinator readiness note.

References: [adr/0025-foundry-prompt-agent-ga-pattern.md](adr/0025-foundry-prompt-agent-ga-pattern.md); [worklog/2026-06-11.md](worklog/2026-06-11.md); [worklog/2026-06-12.md](worklog/2026-06-12.md).

### BUG-0025 — Server-side KB MCP call 401s under the project search connection identity

Area: infra. Severity: high. Status: fixed (found 2026-06-12, fixed 2026-06-12).

Symptom: on the `agent_framework` / Foundry IQ path (`cosmosdb` mode), after BUG-0024's search-MI → OpenAI User grant made the knowledge base's *inner* query-planning model call succeed (a direct REST `/retrieve` with an operator token now returns grounded `Benefit_Options.pdf` content), the end-to-end chat through the backend still returns the out-of-domain fallback with **no `tool`, `reasoning`, or `citation` SSE frames** and **no backend error** (`200 OK`). A standalone orchestrator diagnostic that reproduces the production wiring (credential → FoundryIQ llm → Foundry agents provider → cosmosdb client → `agent_framework` orchestrator) with `openai` + `agent_framework` DEBUG logging exposes the real failure in the Responses-API event stream:

```text
McpListTools(server_label='cwyd-kb', tools=[], type='mcp_list_tools', error=None)
ResponseMcpListToolsInProgressEvent(...)
ChatClientException / openai.APIError: Authentication failed when connecting to the MCP server:
  https://srch-<SUFFIX>.search.windows.net:443/knowledgebases/cwyd-kb/mcp:
  Response status code does not indicate success: 401 (Unauthorized). Verify your authentication headers.
```

The Responses API attempts `mcp_list_tools` against the KB MCP endpoint, gets `401`, and the agent ends up with an empty tool list (`tools=[]`) → no grounding → the out-of-domain refusal. In the live backend the orchestrator's `run()` catches only `AzureError`, so this `ChatClientException` / `openai.APIError` does not surface as a clean SSE `error` frame — it manifests as the refusal answer.

Root cause: the KB retrieval tool is attached **server-side**. `AgentFrameworkOrchestrator._build_kb_tool()` returns `MCPTool(server_url=…/knowledgebases/cwyd-kb/mcp, require_approval="never", allowed_tools=["knowledge_base_retrieve"], project_connection_id=<search connection>)`, serialized via `.as_dict()` into the agent's `extra_tools`. The Responses API therefore opens the MCP connection **itself**, under the *project search connection's* identity (`search-srch-<SUFFIX>`, category `CognitiveSearch`, `authType AAD`, no explicit managed-identity binding — the live connection reports `useWorkspaceManagedIdentity: false`, and `ai-project-search-connection.bicep` creates it with bare `authType: 'AAD'`). That identity is not accepted by the search MCP endpoint: the search service runs `aadOrApiKey` auth with `aadAuthFailureMode: http401WithBearerChallenge`, where a *missing/invalid* token returns `401` (authentication) and an *under-privileged* token returns `403` (authorization). The `401` therefore means no valid bearer reaches the endpoint — the runtime is not presenting the project managed identity's token. The **project** MI *does* hold `Search Index Data Reader` on the search service (sufficient per the Foundry RBAC "retrieve from a knowledge base" row, and assigned 7 days earlier so propagation is ruled out), but it is not the identity the server-side MCP path presents.

This is distinct from BUG-0020 and BUG-0024:

- BUG-0020 fixed the **SDK-side** (client-side) MCP connect via the Agent Framework `header_provider` / default-`Authorization` httpx client, minting the bearer from the *backend's* own credential. The GA pattern (ADR 0025, BUG-0021/0022) moved MCP execution **server-side** (`project_connection_id`), so that hook no longer runs — the connect is now authenticated by Foundry under the connection identity, not by our process.
- BUG-0024 was the KB's **inner** call (knowledge base → query-planning model), authenticated by the *search service* MI. BUG-0025 is the **outer** call (agent → KB MCP endpoint), authenticated by the *project connection* identity. BUG-0024's verification used a direct `/retrieve` with an operator token for the outer call, so it exercised only the inner hop and never surfaced this outer-hop `401`.

Why it was not caught earlier: orchestrator unit tests fake the agent + Responses API entirely, so the server-side `mcp_list_tools` connect is never exercised; this is the first end-to-end agent run against the live KB MCP endpoint with the project connection identity, and the failure is swallowed (the agent refuses rather than erroring).

Root cause confirmed against MACAE (the read-only reference architecture, whose server-side KB MCP grounding works): CWYD points the `MCPTool` at the **wrong category of connection**. The KB MCP tool's `project_connection_id` must reference a dedicated **`RemoteTool` / `ProjectManagedIdentity`** connection that targets the KB MCP URL and pins the token audience — not the general-purpose `CognitiveSearch` / `AAD` connection CWYD reuses. MACAE creates one connection per KB at post-deploy (`infra/scripts/seed_kb_connections.py`) via `PUT …/connections/{kb}-mcp?api-version=2025-04-01-preview` with this payload:

```json
{ "properties": {
    "category": "RemoteTool",
    "target": "https://srch-<SUFFIX>.search.windows.net/knowledgebases/<kb>/mcp?api-version=2025-11-01-preview",
    "authType": "ProjectManagedIdentity",
    "useWorkspaceManagedIdentity": true,
    "isSharedToAll": true,
    "audience": "https://search.azure.com",
    "metadata": { "ApiType": "Azure" } } }
```

The explicit `audience: https://search.azure.com` is the linchpin: it makes Agent Service mint a **project-MI** token for the **search** audience and present it to the MCP endpoint. CWYD's `CognitiveSearch` / bare-`AAD` connection (`useWorkspaceManagedIdentity: false`, no `audience`) mints no usable token for the generic server-side MCP path, so the endpoint sees no valid bearer → `401`, `mcp_list_tools` returns `tools=[]`, and the agent refuses. MACAE keeps a `CognitiveSearch` connection too, but only as the *base* search connection — and even that sets `useWorkspaceManagedIdentity: true` (`modules/aifp-connections.bicep`); the KB tool uses the separate `RemoteTool` connection. ### BUG-0028 — `langgraph` over-refuses in-domain queries (strict CWYD guardrail → gpt-5.1 fallback)

Area: backend. Severity: high. Status: fixed (found 2026-06-12; root cause confirmed by live prompt-variant probes; prompt-softening fix landed + live-verified 2026-06-14).

Symptom: with the active orchestrator set to `langgraph`, a normal in-domain question ("tell me about employee benefits") returns the out-of-domain fallback ("The requested information is not available in the retrieved data. Please try another query or topic.") — a single `answer` frame, no `tool` / `reasoning` / `citation` frames, HTTP 200, no backend error. The same deployment and corpus ground correctly on the `agent_framework` path.

Root cause: the `langgraph` orchestrator does raw chat completion — it retrieves with client-side Azure AI Search, injects the results as a `[docN]` "Sources:" system message, and streams the chat model with the shared CWYD system prompt (`compose_cwyd_instructions(CWYD_DEFAULT_BODY)`). That prompt ends with the non-negotiable `CWYD_GUARDRAIL`, whose grounding clause — "if the retrieved documents do not contain enough information to answer the query, or if no documents are retrieved, your only response is [fallback]" — makes `gpt-5.1` read "enough information" ultra-conservatively and refuse relevant-but-broad queries. Retrieval and message structure are ruled out: direct index search returns 5 relevant hits, and a minimal prompt + the same injected sources grounds reliably. `agent_framework` survives the identical prompt because Foundry IQ executes retrieval server-side and presents the KB results as authoritative tool outputs, which the agent runtime is tuned to ground on; the raw chat-completion path obeys the strict prompt literally.

Confirmation (five live guardrail-variant probes, temp 0.0 + default): the strict prompt refuses the benefits query 8/8 (temperature is not the lever); softening the guardrail grounds most in-domain queries (health plans, handbook, "tell me about the company") while all out-of-domain / creative requests still refuse — but the single broad "tell me about employee benefits" overview sits exactly on gpt-5.1's grounding decision boundary for this small corpus (non-deterministic even at temp 0.0), and a naively permissive guardrail regresses the BUG-0011 grounding guarantee (leaks ungrounded creative content such as "write me a poem about dragons"). So there is a genuine tension: permissive enough to ground broad in-domain queries vs. strict enough to refuse ungrounded creative / out-of-domain requests.

Impact by mode: in `cosmosdb` mode the intended path is `agent_framework` (Foundry IQ), which grounds correctly end-to-end, so `langgraph` is a legacy / fallback path there. In `postgresql` mode `langgraph` is the only supported orchestrator (no pgvector knowledge source for Foundry IQ — ADR 0021 / 0022), so this over-refusal blocks answering entirely for every postgres deployment. This is the "Phase 3 (prompt softening)" work already tracked in the worklog.

Secondary finding (separate code defect): `LLMProvider.complete()` calls `self.chat(messages, deployment=...)` without threading `temperature` / `max_tokens`, so the configured `openai_temperature` (0.0) and `openai_max_tokens` are ignored on the `langgraph` path — the model runs at its default temperature, contributing to the boundary non-determinism. Threading them was attempted as a follow-on unit but **reverted**: passing `max_tokens` made every `langgraph` chat 500 on `gpt-5.1`, which rejects `max_tokens` in favor of `max_completion_tokens` (tracked as BUG-0029). The threading is deferred behind that fix.

Fix (landed 2026-06-14, with operator consent per Hard Rule #10 — edits the non-negotiable, BUG-0011-protected shared CWYD safety prompt): softened the `CWYD_GUARDRAIL` grounding clause and replaced the `CWYD_DEFAULT_BODY` strict v1 in/out-of-domain section ("read sentence by sentence", "only when … enough information", "think twice", "you cannot decide") with a relevance-based "## On answering from the retrieved documents:" section. The new guardrail directive is positive and relevance-anchored — *"When the retrieved documents contain information relevant to the user's question, you **must** answer using that information and cite each claim. Do not refuse just because the documents are brief, fragmentary, or do not cover every detail … Only when none of the retrieved documents are relevant … your only response is [fallback]"* — and explicitly bans ungrounded creative content (*"Never produce creative content (such as stories, poems, jokes, or song lyrics) unless the retrieved documents contain it"*) so the BUG-0011 grounding guarantee is preserved. The exact fixed fallback string, the `[doc+index]` citation format, the no-URLs rule, and the `compose_cwyd_instructions` guardrail-wrapping (BUG-0011 invariant) are all unchanged. Landed test-first: four new invariant assertions in `test_definitions.py` (relevance-based grounding directive present, strict "enough information" wording absent, creative-content ban present, strict v1 body drag dropped) plus the existing fallback/citation/wrapping assertions stay green (16/16). Live-verified on `langgraph` + `gpt-5.1` + `cwyd-index`: "tell me about employee benefits" grounded 3/3 (the prior on-the-boundary query), "what health plans are available?" grounded, "what is the capital of France?" and "write me a poem about dragons" both returned the fixed fallback. Both orchestrators share the prompt, so `agent_framework` inherits the softer grounding while keeping the same safety ban.

### BUG-0031 — Authored-prompt guardrail wrapping applied only on `agent_framework`, not `langgraph`

Both consumers route the **raw** stored override through this one seam, each wrapping it exactly once: `resolve_effective_config` ([services/admin.py](../../src/backend/services/admin.py)) re-resolves `values["cwyd_agent_instructions"]` through the seam after the overlay loop (this is the value `langgraph` injects as its SYSTEM message), and `_resolve_definition` ([base.py](../../src/backend/core/providers/agents/base.py)) calls the same seam instead of an inline `compose_cwyd_instructions`. There is no double-wrap because each consumer wraps the raw override (`overrides.cwyd_agent_instructions`), never the already-wrapped effective value. The three previous composition references (default constant + `_resolve_definition` + the absent langgraph wrap) collapse to one definition. Tests: `test_definitions.py` pins the seam's None/blank/override behavior and that `resolve_cwyd_instructions(None) == CWYD_AGENT.instructions`; `test_services_admin.py` asserts the effective override is guardrail-wrapped (count == 1, body leads) and a blank override falls back to the default; `test_base.py` asserts the agent_framework path equals `resolve_cwyd_instructions(override)`; `test_conversation.py` asserts the router forwards a guardrail-wrapped `system_prompt` (the langgraph-boundary regression). Mechanism:

1. `GET /api/admin/config` ([admin.py](../../src/backend/routers/admin.py)) returns `cwyd_agent_instructions = CWYD_AGENT.instructions`, which is the **wrapped** default — `compose_cwyd_instructions(CWYD_DEFAULT_BODY)` = body + `CWYD_GUARDRAIL`.
2. If the FE admin editor pre-populates the field from that GET, the operator is editing text that already contains the guardrail.
3. On `PATCH /api/admin/config` the RAI gate screens the prompt (passes) and stores it raw — the stored override now contains the guardrail.
4. At runtime the shared `resolve_cwyd_instructions` seam (BUG-0031) wraps the raw override via `compose_cwyd_instructions`, appending `CWYD_GUARDRAIL` a second time. The effective instructions contain two guardrail blocks.

Impact / reachability: only when an operator saves a custom `cwyd_agent_instructions` that includes the guardrail text (the common path if the editor seeds from the wrapped default). Not a safety regression — the guardrail is duplicated, never dropped, and the RAI write-gate still screens authored prompts — so severity is low. The cost is a bloated system prompt (the non-negotiable rules repeated) and a confusing authoring experience (the operator sees and must scroll past the fixed guardrail). Before BUG-0031 the duplication occurred only on `agent_framework` (`_resolve_definition` wrapped) while `langgraph` injected the operator's copy verbatim; after BUG-0031 both paths wrap, making the double-wrap consistent across orchestrators.

Fix direction (not yet landed): make the persona **body** the unit of storage and editing. `GET /api/admin/config` returns `CWYD_DEFAULT_BODY` (or the saved body) — not the wrapped default — so the editor shows and round-trips only the body; the guardrail is applied exactly once at runtime by `resolve_cwyd_instructions`. This changes the admin GET contract and the FE editor's expected value, so it is out of scope for the BUG-0031 seam fix and tracked for a later turn. An alternative (strip a leading/trailing guardrail before re-wrapping) is rejected as brittle string surgery.

References: BUG-0031 (the shared composition seam that made the double-wrap consistent and surfaced this round-trip); BUG-0011 (the original guardrail-wrapping invariant); [worklog/2026-06-14.md](worklog/2026-06-14.md).

### BUG-0034 — Uploaded document reports success but never appears in the documents list

Area: functions. Severity: high. Status: open (found 2026-06-14).

Symptom: an operator uploads a document through the admin Add-data UI; the request returns `200` with an `UploadResponse` (the UI shows success), but the document never appears in the admin documents list (the Delete-data grid), so it cannot be managed or — by implication — used for grounding.

Root cause (not yet diagnosed — hypothesis): the upload endpoint (`POST /api/admin/documents` → `backend.services.ingestion.upload_document`) is a two-step "store blob + enqueue for indexing" operation — it writes the file to the documents container and sends a message to the doc-processing queue, then returns success. It does **not** index the document inline. The documents list is sourced from the Azure AI Search index (`list_sources` reads distinct `title` values), so a file appears there only after the ingestion pipeline consumes the queue message and parses → chunks → embeds → pushes the chunks into the index. The reported success therefore confirms only "blob stored + message enqueued", not "indexed". The break is somewhere between enqueue and index: the Functions ingestion pipeline (Phase 6 — `batch_start` / `batch_push`) may not be running locally, may not be wired to the same queue/container the backend uses, or may be failing to push (poison message). It is also possible this is an environment / run-config gap (the backend runs under `uvicorn` but no Functions host is running to drain the queue) rather than a code defect.

First diagnostic steps: (1) confirm the blob landed in the documents container; (2) confirm a message was enqueued on the doc-processing queue, and that the backend and Functions agree on the queue / container names; (3) confirm a Functions host is running and consumed the message (check queue depth + the poison queue); (4) confirm the Search push ran (query the index for the new `title`). Classify as a code defect vs. an environment gap from there, and reclassify the area to `backend` if the enqueue itself is the gap.

References: BUG-0003 (the same documents list — `list_sources` — which now returns `200` after its facet→paging fix, so the list itself is healthy; this defect is upstream of it, in ingestion); [worklog/2026-06-14.md](worklog/2026-06-14.md).

### BUG-0035 — `agent_framework` sends unsupported `temperature` to the reasoning model (400)

Area: backend. Severity: high. Status: fixed (found 2026-06-14, fixed 2026-06-14).

Symptom: on the BUG-0013 reasoning path (the answer deployment auto-detected as reasoning-capable), forcing the `agent_framework` orchestrator live fails the agent run with `400 - Unsupported parameter: 'temperature' is not supported with this model.`, so the conversation stream emits an `error` frame instead of reasoning + answer.

Root cause: `AgentFrameworkOrchestrator.run()` built `ChatOptions(temperature=…, max_tokens=…)` for every run and passed it to `agent.run(...)`. When the run also requests a Responses-API `reasoning` option, the gpt-5 / o-series answer deployment (`gpt-5.1`) switches to reasoning-model semantics and rejects the `temperature` sampling parameter. The `langgraph` reasoning path never hit this because its `reason()` sends neither `temperature` nor `max_tokens`. The whole `agent_framework` reasoning path was never exercised live before because the deployment runs `langgraph` via a persisted cosmos orchestrator override (`resolve_effective_config` overlays it on the `agent_framework` env default), so only the new both-orchestrator integration test — which sets `app.state.runtime_overrides = RuntimeConfig(orchestrator_name="agent_framework")` after lifespan boot — drove the path and surfaced the 400.

Fix: branch `run()` on the auto-detected reasoning capability (`llm.supports_reasoning()`) — in reasoning mode build a bare `ChatOptions()` and set only the `reasoning` option (omit `temperature`/`max_tokens`), symmetric with the `langgraph` `reason()` path; in non-reasoning mode keep `ChatOptions(temperature, max_tokens)` unchanged. The non-reasoning `agent_framework` path is unaffected — it sends `temperature=0.0` and was live-verified grounding an answer under BUG-0025, so this is strictly a reasoning-mode combination defect, not a broader temperature problem.

Verification: the parametrized `tests/integration/test_conversation_live.py::test_reasoning_feed_streams_substantive_model_reasoning[agent_framework]` now passes live (substantive reasoning streams, no 400); the `langgraph` parameter also passes. Unit coverage: `test_run_requests_reasoning_summary_when_model_supports_it` asserts the `reasoning` option is present and `temperature`/`max_tokens` are absent; `test_run_omits_reasoning_option_when_model_lacks_reasoning` and the default-sampling tests assert the non-reasoning path is unchanged.

References: [worklog/2026-06-14.md](worklog/2026-06-14.md); related BUG-0013 (the reasoning feed this unblocks), BUG-0025 (prior `agent_framework` non-reasoning live verification), BUG-0029 (the sibling `max_tokens`→`max_completion_tokens` chat-path parameter defect).

### BUG-0036 — canned KB-search narration persists alongside real model reasoning

Area: frontend. Severity: low. Status: fixed (found 2026-06-14, fixed 2026-06-14).

Symptom: once BUG-0013 made the answer model stream substantive reasoning, the thinking panel showed the canned "Searching the knowledge base for relevant sources…" sentence *and* the real reasoning together — the placeholder line was never replaced when genuine reasoning began.

Root cause: the during-the-wait narration (added in BUG-0027, emitted upfront by the shared `run_chat` pipeline on the `reasoning` channel) and the orchestrator's own native reasoning frames are both `reasoning`-channel `OrchestratorEvent`s. The FE `append_reasoning` reducer pushed every `reasoning` frame into a single `reasoning[]` list, and `MessageList` joined the whole list, so the placeholder and the model reasoning concatenated. SSE is append-only, so the already-displayed narration could not be removed without FE state support.

Fix: mark the transient narration so the client can drop it. (1) Backend — `run_chat` emits the `retrieval_hint` frame with `metadata={"placeholder": True}`; `format_sse` already forwards `metadata` for every channel, so it reaches the FE as `ev.metadata.placeholder === true`. (2) FE model — `ChatMessage` gains a `reasoningPlaceholder?: string` slot, held apart from `reasoning[]`. (3) FE reducer — a new `set_reasoning_placeholder` action sets (never appends) the placeholder. (4) FE handler — `MessageInput` routes a `reasoning` frame with `metadata.placeholder === true` to `set_reasoning_placeholder`, all other `reasoning` frames to `append_reasoning`. (5) FE renderer — `MessageList` shows the panel when streaming, or when `reasoning` is non-empty, or when a non-empty placeholder exists, and renders `reasoning.join("")` when non-empty else `reasoningPlaceholder` — so the first real reasoning frame supersedes the placeholder, and a non-reasoning model keeps the placeholder as the sole content (no regression).

Verification: `tests/backend/core/pipelines/test_chat.py` asserts the narration frame carries `metadata == {"placeholder": True}` and the orchestrator's native reasoning frame does not (18/18). FE Vitest: `ChatContext.test.tsx` (placeholder set without touching `reasoning`, replace-not-append), `MessageInput.test.tsx` (placeholder frame → `reasoningPlaceholder`, subsequent real frame → `reasoning[]`), `MessageList.test.tsx` (placeholder shown when no reasoning yet; real reasoning drops the placeholder) — 87/87 across the three suites. `tsc --noEmit` clean on the frontend source + test workspaces; `pyright` 0/0/0 on `chat.py`.

References: [worklog/2026-06-14.md](worklog/2026-06-14.md); related BUG-0013 (the reasoning content this polishes), BUG-0027 (the upfront narration whose lifecycle this completes).
